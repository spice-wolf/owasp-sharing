# 一、`OWASP_zh.pdf` 重点注意事项

## 1.1 这份材料的定位

- 这不是某个语言或框架的“操作手册”，而是一份适合放进研发流程和编码规范的“安全底线清单”。
- 虽然文档较早，但里面大量原则并不过时，尤其适合团队内部分享时做“底座”：
  - 输入必须验证
  - 输出必须按上下文编码
  - 认证、会话、授权必须在服务端收口
  - 加密、日志、配置、数据库、文件上传都要有最小安全基线
- 可以把它理解为：`不管技术怎么变，Web 开发总要守住的通用安全常识`

## 1.2 适合在分享里强调的总原则

### 原则 1：永远不要把“客户端限制”当成安全控制

- PDF 明确强调：前端校验、隐藏字段、下拉框、单选框都只能提升体验，不能代替服务端校验。
- 对我们团队尤其重要：
  - 前端做了字段限制，不代表后端就安全
  - 攻击者完全可以改请求体、URL、Header、Cookie
  - 抓包、重放、脚本构造请求的成本很低

可结合的现实例子：

- 前端把 `role=admin` 隐藏了，但接口仍按传入字段更新
- 前端不展示“导出全部用户”按钮，但接口未做权限校验
- 页面限制上传 `.jpg`，但后端只看扩展名

### 原则 2：安全控制必须尽量集中化、服务端化、可复用

- PDF 多次强调：
  - 集中的输入验证规则
  - 集中的认证控制
  - 集中的授权控制
  - 集中的日志记录控制
- 对 Spring Boot 团队的意义：
  - 不要把安全逻辑散在 Controller 里到处复制
  - 优先使用统一过滤器、拦截器、中间件、网关、Spring Security、统一异常处理、统一审计框架
  - 统一策略比“每个接口开发者自己记得写”更可靠

### 原则 3：默认拒绝，而不是默认放行

- 访问控制、异常处理、配置、文件访问、服务访问，都应倾向于 `deny by default`
- 团队里最容易出问题的地方：
  - 新增接口忘了鉴权
  - 新增对象存储桶默认公开
  - 新增管理端点默认暴露
  - 异常场景“临时放过”导致 fail-open

### 原则 4：安全问题往往不是“某一行代码错了”，而是“全链路没收口”

- 文档覆盖了代码、数据库、配置、传输、日志、文件、部署
- 这很适合给跨角色团队讲：
  - 前端只管 XSS 不够
  - 后端只管 SQL 注入不够
  - 运维只管打补丁不够
  - 需要研发、测试、运维、平台一起形成闭环

## 1.3 最值得讲的章节与团队场景映射

## 输入验证

### PDF 核心点

- 所有来自不可信源的数据都要校验
- 服务端执行校验
- 校验类型、长度、范围、字符集
- 优先白名单
- 规范化后再校验
- 对参数、URL、Header、Cookie 都要校验

### PDF 原文摘录

> “在可信系统（比如：服务器）上执行所有的数据验证。”
>
> “识别所有的数据源，并将其分为可信的和不可信的。验证所有来自不可信数据源（比如：数据库，文件流，等）的数据。”
>
> “应当为应用程序应提供一个集中的输入验证规则。”
>
> “验证正确的数据类型。验证数据范围。验证数据长度。”
>
> “尽可能采用‘白名单’形式，验证所有的输入。”
>
> “在处理以前，验证所有来自客户端的数据，包括：所有参数、URL、HTTP 头信息（比如：cookie 名字和数据值）。”

### 对团队最相关的落地理解

- 输入不只是表单字段，还包括：
  - REST 参数
  - JSON body
  - 上传文件名
  - Excel 导入内容
  - MQ 消息
  - Redis 中读出来的值
  - 第三方回调参数
  - 数据库里历史脏数据
- “来自数据库的数据也可能不可信”这一点很适合强调
  - 因为被污染的数据以后仍可能被再次渲染或拼接

### 可讲的团队例子

- 订单状态接口只校验“非空”，没校验枚举范围，导致越权流转
- 导出报表接口接受排序字段名，拼入 SQL `order by`
- 文件下载接口接受文件路径参数，出现路径穿越
- 回调接口对来源系统字段不校验，被伪造调用

### 对应建议

- Spring Boot DTO 层使用 `@NotNull`、`@Size`、`@Pattern`、`@Min/@Max`
- 业务层继续校验“业务合法性”，不要只靠注解
- 对排序字段、字段名、状态机流转使用白名单映射
- 对上传文件名、对象 key、导出模板名等做强约束

## 输出编码

### PDF 核心点

- 按上下文输出编码
- 不是“统一转义一次”就够
- HTML、URL、JavaScript、SQL、LDAP、OS 命令上下文不同

### PDF 原文摘录

> “在可信系统（比如：服务器）上执行所有的编码。”
>
> “为每一种输出编码方法采用一个标准的、已通过测试的规则。”
>
> “通过语义输出编码方式，对所有返回到客户端的来自于应用程序信任边界之外的数据进行编码。”
>
> “针对 SQL、XML 和 LDAP 查询，语义净化所有不可信数据的输出。”
>
> “对于操作系统命令，净化所有不可信数据输出。”

### 对团队最相关的落地理解

- 这部分是前端、后端都容易误判的点
- 核心不是“过滤脏字符”，而是“在正确的输出位置做正确的编码”
- 典型误区：
  - 以为后端把 `<` `>` 替换掉就一劳永逸
  - 以为 React/Vue 默认转义就不会有 XSS
  - 忘记 `v-html`、富文本、Markdown 渲染、模板拼接、URL 拼接

### 可讲的团队例子

- 管理后台评论内容支持富文本，展示时未做白名单清洗
- 报表页面把用户输入直接回显到 HTML 属性或 JS 片段中
- 下载链接把文件名未经编码直接拼进响应头，导致响应拆分风险

### 对应建议

- 前端默认使用框架的安全渲染能力，少用原始 HTML 注入
- 富文本统一引入可信清洗库
- 后端输出到不同上下文前选择对应编码策略
- 任何需要拼 shell、SQL、模板表达式的地方，都优先改成“结构化 API”

## 身份验证和密码管理

### PDF 核心点

- 除公开资源外，应要求身份验证
- 使用集中式认证服务
- 密码做强哈希，不要用 MD5
- 登录失败提示不要暴露过多信息
- 要有复杂度、长度、锁定、多因子、密码重置控制

### PDF 原文摘录

> “除了那些特定设为‘公开’的内容以外，对所有的网页和资源要求身份验证。”
>
> “在任何可能的情况下，建立并使用标准的、已通过测试的身份验证服务。”
>
> “为所有身份验证控制使用一个集中实现的方法，其中包括利用库文件请求外部身份验证服务。”
>
> “如果您的应用程序管理着凭证的存储，那么应当保证只保存了通过使用强加密单向 salted 哈希算法得到的密码，并且只有应用程序具有对保存密码和密钥的表/文件的写权限（如果可以避免的话，不要使用 MD5 算法）。”
>
> “身份验证的失败提示信息应当避免过于明确。”
>
> “当连续多次登录失败后（比如：通常情况下是 5 次），应强制锁定账户。”
>
> “为高度敏感或重要的交易账户使用多因子身份验证机制。”

### 对团队最相关的落地理解

- 这部分在内部系统里最容易被“因为是内网”而弱化
- 现实风险往往来自：
  - 弱密码
  - 默认账号
  - 管理员密码长期不改
  - 测试环境账号复用到生产
  - 登录接口无频控
  - 找回密码逻辑过弱

### Java/Spring Boot 场景建议

- 使用成熟认证框架而不是自造轮子
- 密码存储使用强哈希算法
- 管理台、运维台、配置中心、跳板类能力优先 MFA
- 登录、验证码、短信、邮箱找回流程都要纳入安全设计
- JWT 不应当被当作“万能安全方案”

## 会话管理

### PDF 核心点

- 会话 ID 由服务端生成
- 登录后切换新会话
- 注销必须真正失效
- 不在 URL、错误信息、日志里暴露会话标识
- Cookie 加 `Secure`、`HttpOnly`

### PDF 原文摘录

> “使用服务器或者框架的会话管理控制。应用程序应当只识别有效的会话标识符。”
>
> “会话标识符必须总是在一个可信系统（比如：服务器）上创建。”
>
> “注销功能应当完全终止相关的会话或连接。”
>
> “如果一个会话在登录以前就建立，在成功登录以后，关闭该会话并创建一个新的会话。”
>
> “不要在 URL、错误信息或日志中暴露会话标识符。会话标识符应当只出现在 HTTP cookie 头信息中。”
>
> “为在 TLS 连接上传输的 cookie 设置‘安全’属性。”
>
> “将 cookie 设置为 HttpOnly 属性，除非在应用程序中明确要求了客户端脚本程序读取或者设置 cookie 的值。”

### 对团队最相关的落地理解

- 很适合连接现在的 `Session`、`JWT`、`SSO`
- 关键不是“用哪一种”，而是：
  - 会不会被窃取
  - 会不会长期有效
  - 能不能撤销
  - 登出后是否真的失效

### 可讲的团队例子

- JWT 过期时间过长，用户离职后仍可调用接口
- Token 出现在 URL 查询参数里，被日志和代理记录
- 切换账号后旧 token 仍有效
- Remember-me 机制没有额外约束

## 访问控制

### PDF 核心点

- 权限判断必须基于服务端可信对象
- 每个请求都要校验授权
- 保护 URL、功能、对象、数据、配置
- 只把 referer 当补充，不能单独作为认证依据

### PDF 原文摘录

> “只使用可信系统对象（比如：服务器端会话对象）以做出访问授权的决定。”
>
> “使用一个单独的全站点部件以检查访问授权。”
>
> “在每个请求中加强授权控制，包括：服务器端脚本产生的请求，‘includes’和来自象 AJAX 和 FLASH 那样的富客户端技术的请求。”
>
> “限制只有授权的用户才能访问受保护的 URL。”
>
> “限制只有授权的用户才能访问直接对象引用。”
>
> “服务器端执行的访问控制规则和表示层实施的访问控制规则必须匹配。”
>
> “仅使用‘referer’头作为补偿性质的检查，它永远不能被单独用来进行身份验证检查，因为它可以被伪造。”

### 对团队最相关的落地理解

- 这是对 Web 系统最有现实意义的一章之一
- 需要突出三种常见错误：
  - 只做“是否登录”，没做“是否有权”
  - 只做页面权限，没做接口权限
  - 只做功能权限，没做数据权限

### 典型场景

- 用户可通过修改 `userId/orderId/projectId` 查看他人数据
- 管理页面不显示按钮，但直接调用接口仍能成功
- 多租户系统没有真正按租户隔离数据
- 导出接口直接返回跨部门数据

### 对应建议

- RBAC 只是起点，很多场景需要 `RBAC + 数据归属 + 业务规则`
- 在 Service/Domain 层做关键授权判断，而非只在前端菜单控制
- 对关键对象读取、修改、删除做 ownership 校验
- 增加越权专项测试

## 加密规范

### PDF 核心点

- 加密操作放在可信系统
- 使用已验证随机数生成器
- 做好密钥管理

### PDF 原文摘录

> “所有用于保护来自应用程序用户秘密信息的加密功能都必须在一个可信系统（比如：服务器）上执行。”
>
> “保护主要秘密信息免受未授权的访问。”
>
> “安全的处理加密模块失败的操作。”
>
> “为防范对随机数据的猜测攻击，应当使用加密模块中已验证的随机数生成器生成所有的随机数、随机文件名、随机 GUID 和随机字符串。”
>
> “建立并使用相关的政策和流程以实现加、解密的密钥管理。”

### 对团队最相关的落地理解

- 真正的难点通常不是“有没有加密”，而是：
  - 算法是否过时
  - 密钥如何保管
  - 是否硬编码
  - 是否轮换
  - 是否把“可逆加密”错误用于密码存储

### 可讲的现实例子

- `application.yml` 里明文数据库密码
- JWT 密钥写死在仓库
- 证书、私钥、AK/SK 在开发群里传播
- 把手机号、身份证只做 Base64 处理就当“加密”

## 错误处理和日志

### PDF 核心点

- 不暴露堆栈、会话、账号等敏感信息
- 使用通用错误消息
- 记录安全事件
- 日志里不能保存敏感信息

### PDF 原文摘录

> “不要在错误响应中泄露敏感信息，包括：系统的详细信息、会话标识符或者帐号信息。”
>
> “使用错误处理以避免显示调试或堆栈跟踪信息。”
>
> “使用通用的错误消息并使用定制的错误页面。”
>
> “所有的日志记录控制应当在可信系统（比如：服务器）上执行。”
>
> “日志记录控制应当支持记录特定安全事件的成功或者失败操作。”
>
> “不要在日志中保存敏感信息，包括：不必要的系统详细信息、会话标识符或密码。”
>
> “记录所有失败的输入验证。记录所有的身份验证尝试，特别是失败的验证。记录所有失败的访问控制。”

### 对团队最相关的落地理解

- 很适合给后端和运维一起讲
- 常见问题：
  - Spring Boot 默认错误页暴露细节
  - 日志打出 token、手机号、身份证、SQL、请求体
  - 只有报错日志，没有安全审计日志
  - 有日志但没人看、没人告警

### 可讲的现实例子

- `Exception` 直接返回给前端，暴露类名、表名、路径
- 订单支付失败没有审计链路，事后无法复盘
- 登录失败只记“失败”，不记来源 IP、账号、原因

## 数据保护

### PDF 核心点

- 最小权限
- 敏感数据不要明文存储
- 不要在 GET 参数里放敏感信息
- 禁止客户端缓存敏感页面

### PDF 原文摘录

> “授予最低权限，以限制用户只能访问为完成任务所需要的功能、数据和系统信息。”
>
> “即使在服务器端，任然要加密存储的高度机密信息，比如，身份验证的验证数据。”
>
> “不要在客户端上以明文形式或其他非加密安全模式保存密码、连接字符串或其他敏感信息。”
>
> “不要在 HTTP GET 请求参数中包含敏感信息。”
>
> “禁止客户端缓存网页，因为可能包含敏感信息。‘Cache-Control: no-store’，可以和 HTTP 报头控制‘Pragma: no-cache’一起使用。”

### 团队场景

- URL 中带 token、手机号、工号、重置口令
- 导出文件长期放在公开 URL
- 临时文件、缓存、对象存储留存敏感数据
- 测试库复制了生产脱敏前数据

## 通讯安全

### PDF 核心点

- 敏感信息使用 TLS
- 证书必须有效
- 失败不能回退到不安全连接

### PDF 原文摘录

> “为所有敏感信息采用加密传输。其中应该包括使用 TLS 对连接的保护，以及支持对敏感文件或非基于 HTTP 连接的不连续加密。”
>
> “TLS 证书应当是有效的，有正确且未过期的域名，并且在需要时，可以和中间证书一起安装。”
>
> “没有成功的 TLS 连接不应当后退成为一个不安全的连接。”
>
> “为所有要求身份验证的访问内容和所有其他的敏感信息提供 TLS 连接。”
>
> “为包含敏感信息或功能、且连接到外部系统的连接使用 TLS。”

### 团队场景

- 内部服务因为“内网可信”而走明文
- 回调接口校验证书不严
- 证书过期监控缺失
- 网关到后端、后端到中间件之间缺乏一致传输保护

## 系统配置

### PDF 核心点

- 服务器、框架、组件保持最新认可版本
- 打补丁
- 删除不需要的功能和文件
- 关闭目录列表
- 移除多余 HTTP 方法、版本信息

### PDF 原文摘录

> “确保服务器、框架和系统部件采用了认可的最新版本。”
>
> “确保服务器、框架和系统部件安装了当前使用版本的所有补丁。”
>
> “关闭目录列表功能。”
>
> “移除所有不需要的功能和文件。”
>
> “在部署前，移除测试代码和产品不需要的功能。”
>
> “禁用不需要的 HTTP 方法，比如 WebDAV 扩展。”
>
> “移除在 HTTP 相应报头中有关 OS、Web 服务版本和应用程序框架的无关信息。”

### 团队场景

- Spring Boot Actuator 暴露过多端点
- Nginx/Tomcat 暴露版本
- 测试接口、Swagger、默认账号在生产可见
- 容器镜像包含不必要工具与样例文件

## 数据库安全

### PDF 核心点

- 参数化查询
- 最小权限
- 连接串不要硬编码
- 关闭不必要数据库功能

### PDF 原文摘录

> “使用强类型的参数化查询方法。”
>
> “当应用程序访问数据库时，应使用尽可能最低的权限。”
>
> “连接字符串不应当在应用程序中硬编码。连接字符串应当存储在一个可信服务器的独立配置文件中，并且应当被加密。”
>
> “删除或者修改所有默认的数据库管理员密码。”
>
> “关闭所有不必要的数据库功能（比如：不必要的存储过程或服务、应用程序包、仅最小化安装需要的功能和选项（表面范围缩减））。”
>
> “应用程序应当以不同的凭证为每个信任的角色（比如：用户、只读用户、访问用户、管理员）连接数据库。”

### 团队场景

- 手写 SQL 拼接
- MyBatis `${}` 滥用
- 报表排序、筛选、表名动态拼接
- 应用使用数据库管理员权限

## 文件管理

### PDF 核心点

- 上传前认证
- 只允许业务需要的类型
- 检查文件头，不只看扩展名
- 文件与 Web 根隔离
- 上传目录禁执行
- 扫描恶意文件

### PDF 原文摘录

> “在允许上传一个文档以前进行身份验证。”
>
> “只允许上传满足业务需要的相关文档类型。”
>
> “通过检查文件报头信息，验证上传文档是否是所期待的类型。只验证文件类型扩展是不够的。”
>
> “不要把文件保存在与应用程序相同的 Web 环境中。文件应当保存在内容服务器或者数据库中。”
>
> “防止或限制上传任意可能被 Web 服务器解析的文件。”
>
> “关闭在文件上传目录的运行权限。”
>
> “对用户上传的文件扫描进行病毒和恶意软件。”

### 团队场景

- 头像上传可上传脚本/木马
- 模板导入可导致公式注入、宏风险
- 文件下载接口路径拼接
- OSS/MinIO 对象桶设置错误导致公开读取

## 通用编码规范

### PDF 核心点

- 尽量使用成熟 API
- 避免把用户数据交给动态执行能力
- 审核第三方库
- 通过更新、签名、校验保护代码完整性

### PDF 原文摘录

> “为常用的任务使用已测试且已认可的托管代码，而不创建新的非托管代码。”
>
> “使用特定任务的内置 API 以执行操作系统的任务。不允许应用程序直接将代码发送给操作系统，特别是通过使用应用程序初始的命令 shell。”
>
> “使用校验和或哈希值验证编译后的代码、库文件、可执行文件和配置文件的完整性。”
>
> “不要将用户提供的数据传递给任何动态运行的功能。”
>
> “审核所有从属的应用程序、第三方代码和库文件，以确定业务的需要，并验证功能的安全性，因为它们可能产生新的漏洞。”
>
> “执行安全更新。如果应用程序采用自动更新，则为您的代码使用加密签名，以确保的您的下载客户端验证这些签名。”

### 团队场景

- 调用 `Runtime.exec`、脚本引擎、表达式引擎
- 动态 Groovy/SpEL 使用不当
- 低质量依赖、无人维护依赖长期存在

## 1.4 最适合我们团队强调的“十条安全编码底线”

1. 所有输入都不可信，服务端统一校验。
2. 所有输出都按上下文编码，特别是 HTML、JS、URL、响应头。
3. 所有权限控制都必须在服务端完成，且要校验到“数据归属”。
4. 登录、会话、Token、登出、重置密码必须作为完整链路设计。
5. 任何 SQL、命令、模板、表达式拼接都要高度警惕注入。
6. 敏感数据、密钥、证书、连接串不得明文散落在代码和配置中。
7. 文件上传、下载、导出、导入必须当成高风险功能设计。
8. 错误信息对外收敛，日志对内完整，但日志里不能泄露敏感数据。
9. 安全配置要自动化、基线化，而不是靠人工记忆。
10. 安全不是“上线前扫一次”，而是需求、设计、编码、测试、部署全过程。

## 1.5 讲这一部分时可用的观点

- 这份 PDF 老，但老得很“底层”，像交通法规，不像某个版本的 API 文档。
- 里面很多内容今天已经可以被框架、网关、平台工具“工程化”实现，这反而说明 OWASP 的价值不是过时，而是被验证了。
- 分享时不建议逐条照念，更适合按“我们日常最容易踩的坑”重组。

# 二、OWASP Top 10:2025 材料收集

## 2.1 已核实的官方结论

- 截至 `2026-05-26`，OWASP 官方已经提供 `OWASP Top 10:2025` 页面。
- 官方 2025 排名如下：
  - `A01:2025 Broken Access Control`
  - `A02:2025 Security Misconfiguration`
  - `A03:2025 Software Supply Chain Failures`
  - `A04:2025 Cryptographic Failures`
  - `A05:2025 Injection`
  - `A06:2025 Insecure Design`
  - `A07:2025 Authentication Failures`
  - `A08:2025 Software or Data Integrity Failures`
  - `A09:2025 Security Logging & Alerting Failures`
  - `A10:2025 Mishandling of Exceptional Conditions`

## 2.2 一个值得讲的总体变化

- 相比很多团队还停留在 2021 版本，2025 版有几个非常值得强调的新信号：
  - `安全配置错误` 上升到第 2 位
  - `软件供应链失败` 上升到第 3 位
  - `异常条件处理不当` 作为新类别进入 Top 10
- 这说明 OWASP 对现代研发现实有了更强回应：
  - 不只是“写错代码”
  - 还包括“配错系统”“依赖失控”“流水线失守”“异常没兜住”

## 2.3 每一项的原始素材

## A01:2025 Broken Access Control

### 是什么

- 用户能够执行本不该执行的操作，访问本不该访问的资源或数据。

### 官方提到的典型问题

- 修改 URL、参数、请求体绕过控制
- 直接对象引用导致查看/修改他人数据
- POST/PUT/DELETE 缺少访问控制
- JWT/Cookie/隐藏字段被篡改
- CORS 配置错误

### 我们团队最容易遇到的场景

- 通过改 `id` 查看他人订单、工单、报表
- 只校验“是否登录”，不校验“是否属于当前租户/部门/项目”
- 管理员接口只在前端隐藏按钮
- 导出、下载、审批类接口遗漏权限校验

### 防范要点

- 默认拒绝
- 服务端统一授权
- 同时控制“功能权限”和“数据权限”
- 对关键资源做 ownership 校验
- 日志记录权限失败并告警
- 做越权专项测试

## A02:2025 Security Misconfiguration

### 是什么

- 系统、应用、云服务从安全角度配置不正确。

### 官方提到的典型问题

- 缺少安全加固
- 不必要功能、服务、页面、权限仍开启
- 默认账号/默认密码未修改
- 详细错误信息暴露
- 服务器/框架/数据库安全设置未收紧
- 缺少安全头

### 我们团队最容易遇到的场景

- Spring Boot Actuator 暴露太多端点
- Swagger/测试页面在生产可见
- 对象存储桶默认公开
- Nginx/Tomcat/网关暴露版本
- 默认镜像、默认密码、默认开放端口
- `server.error.include-stacktrace` 之类配置不当

### 防范要点

- 建立基线化、自动化的加固流程
- 环境配置尽量同构，凭据分离
- 关闭无用功能、样例、目录列表、危险方法
- 安全头统一下发
- 配置检查自动化

## A03:2025 Software Supply Chain Failures

### 是什么

- 软件构建、分发、更新链路中的失控或被攻陷。

### 官方提到的典型问题

- 不跟踪直接与传递依赖
- 组件过期、无维护、已知有漏洞
- 不订阅漏洞公告
- 供应链变更缺少记录
- 流水线权限过大、职责不分离
- 使用不可信来源组件
- 补丁节奏过慢

### 我们团队最容易遇到的场景

- Maven 依赖和传递依赖无人盘点
- 开发机、IDE 插件、代码仓库、制品库安全没人统一管
- 生产包不是同一个构建产物，环境间重新打包
- CI/CD 机器权限太大、secret 到处可读
- 制品未经签名和来源校验

### 防范要点

- 建 SBOM
- 持续盘点依赖版本和漏洞
- 仅从可信源获取组件
- 关键系统启用 MFA、分支保护、最小权限
- 对构建、发布、制品、镜像做签名与来源证明
- 使用灰度/金丝雀发布降低供应链投毒爆炸半径

## A04:2025 Cryptographic Failures

### 是什么

- 缺失加密、加密强度不够、密钥泄露、随机性不足、证书校验不当等问题。

### 官方提到的典型问题

- 旧算法或弱算法
- 默认密钥/弱密钥/密钥复用
- 密钥进代码仓库
- 未强制 TLS
- 证书链校验不当
- IV/nonce 使用不当
- MD5/SHA1 等过时算法

### 我们团队最容易遇到的场景

- 配置文件里明文密码和 key
- JWT secret 写死
- 敏感数据未分类，不知道哪些必须加密
- 内部服务明文传输
- Redis/对象存储/备份未加密
- 密码存储方式老旧

### 防范要点

- 数据分级分类
- 敏感数据传输与静态存储加密
- 密钥进 KMS/HSM 或受控密钥体系
- 密码强哈希
- 强制 TLS 1.2+
- 禁止弱算法和弱随机数

## A05:2025 Injection

### 是什么

- 不可信输入被解释器当作命令/语句的一部分执行。

### 官方提到的典型问题

- SQL、NoSQL、OS Command、LDAP、ORM、EL/OGNL 等注入
- 用户输入未验证、未过滤、未隔离
- 动态查询或字符串拼接
- 还特别提到：LLM 场景中已有相关的 Prompt Injection 风险

### 我们团队最容易遇到的场景

- MyBatis `${}` 动态拼接
- HQL/JPQL/动态 SQL 拼接
- 导出排序字段、过滤表达式、表名动态传入
- 文件处理或诊断工具拼 shell
- SpEL、模板表达式、脚本执行能力使用不当

### 防范要点

- 参数化查询优先
- 输入白名单
- 尽量避免解释器式接口
- SAST/DAST/IAST 与 fuzzing 进入流水线
- 对“结构性输入”单独治理

## A06:2025 Insecure Design

### 是什么

- 控制本身没有被设计出来，或者设计无效。

### 官方强调点

- 区分“设计问题”和“实现问题”
- 需要威胁建模
- 要把安全要求写进需求和用户故事
- 关注业务逻辑与失败路径

### 我们团队最容易遇到的场景

- 密码找回靠安全问题
- 秒杀/抢购/批量占用座位等业务没有反机器人设计
- 审批流、退款流、优惠叠加流没有防滥用约束
- 多租户隔离设计先天薄弱

### 防范要点

- 在需求和设计阶段引入安全审视
- 关键流程做 threat modeling
- 每层增加合理性校验
- 为误用场景设计防线，而不是只为正常流程设计

## A07:2025 Authentication Failures

### 是什么

- 系统把“不该认的人”认成了合法用户，或会话/token 管理失效。

### 官方提到的典型问题

- 凭证填充、密码喷洒、暴力破解
- 默认或弱密码
- 已泄露密码仍可注册
- 弱找回密码流程
- 明文/弱哈希密码存储
- MFA 缺失或回退过弱
- URL 中暴露会话 ID
- 登出/过期处理不正确

### 我们团队最容易遇到的场景

- 管理后台无 MFA
- 短信/邮箱找回密码链路过弱
- SSO token 退出后仍可用
- 会话时间过长
- 异常设备登录缺少风险控制

### 防范要点

- MFA
- 密码黑名单和泄露库校验
- 频控、锁定、风控
- 安全登出与 token 撤销
- 统一认证平台

## A08:2025 Software or Data Integrity Failures

### 是什么

- 没有验证软件、代码、数据工件的来源与完整性。

### 官方提到的典型问题

- 信任了不可信插件、库、模块、CDN
- CI/CD 不验证制品完整性
- 自动更新没有充分验证
- 不安全反序列化/数据篡改

### 我们团队最容易遇到的场景

- 前端静态资源从外部 CDN 拉取且缺少校验
- 制品仓库或镜像仓库控制松散
- 反序列化历史包袱
- 回调数据、缓存数据、状态对象默认被信任

### 防范要点

- 数字签名/完整性校验
- 可信仓库
- CI/CD 分权和审查
- 禁止从不可信客户端接收未签名序列化数据

## A09:2025 Security Logging & Alerting Failures

### 是什么

- 没有记录、监控和告警足够的安全事件，或者日志本身不安全。

### 官方提到的典型问题

- 登录、失败登录、高价值操作未记录
- 错误日志不清晰
- 日志可被篡改
- 日志只在本地
- 缺少阈值、告警、响应升级
- 日志中泄露敏感信息
- 日志编码不当反而被攻击

### 我们团队最容易遇到的场景

- 有业务日志，没有安全审计
- 有日志平台，没有规则和告警
- 登录失败不计数、不报警
- 大量假阳性导致真正事件被淹没

### 防范要点

- 安全事件标准化记录
- 审计日志集中化
- 告警场景与剧本配套
- 日志防篡改与长期留存
- 不记录不该记录的敏感数据

## A10:2025 Mishandling of Exceptional Conditions

### 是什么

- 程序没有正确预防、识别、处理异常条件，导致崩溃、异常状态、失败开放、逻辑漏洞或安全问题。

### 官方强调点

- 这是 2025 新增类别
- 包括错误处理不当、逻辑错误、fail-open、缺参处理错误、空指针、权限不足处理不当等

### 我们团队最容易遇到的场景

- 参数缺失后走默认值，结果默认值权限更大
- 调用下游失败后进入错误兜底，却错误放行
- 风控/鉴权组件超时后选择“先放过”
- 状态机在并发或异常情况下出现非法跳转

### 防范要点

- 在异常发生点就捕获并处理
- 全局异常处理只是最后一道兜底
- 关键控制必须 fail-secure
- 对网络、权限、资源不足、超时、空值、并发异常做专门测试

## 2.4 最适合结合团队技术栈讲的 Top 10 高危场景

- `Spring Boot + MyBatis/MySQL`
  - SQL 注入
  - 越权
  - 异常暴露
  - 数据库账号权限过大
- `Spring Security / SSO / JWT`
  - 认证失败
  - token 生命周期和撤销策略不足
  - 角色与数据权限脱节
- `对象存储 / 文件上传 / 报表导入导出`
  - 文件管理
  - 配置错误
  - 敏感数据泄露
- `CI/CD / Maven / Docker / 制品库`
  - 软件供应链
  - 完整性问题
- `网关 / 反向代理 / Nginx / 云资源`
  - 安全配置错误
  - TLS 与安全头
  - 暴露面过大

## 2.5 这一部分分享时的表达建议

- 不建议按 1 到 10 机械念完。
- 更好的讲法是：
  - 哪些是“我们最容易真遇到的”
  - 哪些是“2025 新趋势”
  - 哪些是“跨前后端运维都相关的”
- 可以把 Top 10 分成四类：
  - 代码类：`A05 A10`
  - 身份与权限类：`A01 A07`
  - 工程与平台类：`A02 A03 A08 A09`
  - 架构与基础控制类：`A04 A06`

# 三、AI 时代 OWASP 的变化与新方向

## 3.1 一个总判断

- OWASP 正在从“经典 Web 应用安全知识库”扩展为“覆盖现代软件与 AI 系统安全生命周期的开放体系”。
- 这种变化不只是多了几份文档，而是范围明显扩大：
  - 从 Web 应用到 LLM/GenAI/Agentic AI
  - 从编码缺陷到供应链、治理、身份、协议层
  - 从开发者指南到可执行工具、指南、对照框架和演练材料

## 3.2 变化一：OWASP Top 10 for LLM 已升级为 OWASP GenAI Security Project

### 已核实的官方信息

- `2025-03-26`，OWASP 宣布原有 `OWASP Top 10 for LLM` 已升级为 `OWASP Gen AI Security Project`，并成为 Flagship 项目。

### 这意味着什么

- OWASP 不再只把 AI 安全理解为“给 LLM 做一个 Top 10”
- 而是开始建设一个更完整的 AI 安全体系：
  - 风险清单
  - 治理清单
  - 事件与威胁情报
  - Agentic 安全
  - AI 红队
  - 安全采用方法

### 分享中可讲的观点

- 这说明 AI 安全已经从“新鲜话题”变成了“需要长期治理的方法论”

## 3.3 变化二：OWASP LLM Top 10 已经更新到 2025 版

### 2025 版列表

- `LLM01:2025 Prompt Injection`
- `LLM02:2025 Sensitive Information Disclosure`
- `LLM03:2025 Supply Chain`
- `LLM04:2025 Data and Model Poisoning`
- `LLM05:2025 Improper Output Handling`
- `LLM06:2025 Excessive Agency`
- `LLM07:2025 System Prompt Leakage`
- `LLM08:2025 Vector and Embedding Weaknesses`
- `LLM09:2025 Misinformation`
- `LLM10:2025 Unbounded Consumption`

### 值得强调的变化信号

- OWASP 已经把“提示词注入”单独视为核心风险
- 不再只盯模型本身，而是把以下内容纳入风险视野：
  - 向量库与 embedding
  - 代理权限过大
  - 输出处理不当
  - 成本与资源失控
  - 模型供应链

### 对我们团队的启发

- 即使我们不是在训练模型，只要我们未来开始接入 LLM：
  - 做智能客服
  - 做代码助手
  - 做自动化审批/问答/文档处理
  - 做 RAG 检索
  - 做 Agent
  - 就已经落入这些风险范畴

## 3.4 变化三：OWASP 开始系统推进 Agentic Security

### 已核实的官方信息

- OWASP GenAI Security Project 已有 `Agentic Security Initiative`
- 官方站点已有 `OWASP Top 10 for Agentic Applications for 2026`

### 这说明什么

- OWASP 认为 Agent 风险不能只用传统 Web 风险或普通 LLM 风险解释
- 因为 Agent 具备：
  - 目标分解能力
  - 工具调用能力
  - 文件系统/网络/执行环境访问能力
  - 持久记忆
  - 跨步骤自治

### 可讲的观点

- 从“会回答”到“会行动”，风险性质发生了变化
- AI 不再只是生成文本，而是在“参与业务流程”和“影响现实系统”

## 3.5 变化四：OWASP Agentic Skills Top 10 的出现

### 已核实的官方信息

- `OWASP Agentic Skills Top 10` 已上线官方项目页
- 它覆盖的技能生态包括：
  - `OpenClaw`
  - `Claude Code`
  - `Cursor/Codex`
  - `VS Code`

### 这个项目为什么很值得在分享里讲

- 它抓住了 AI Agent 的一个很现实的新攻击面：`skill`
- 技能不是单个工具，而是“指导 AI 如何串联工具、完成任务的行为模板”
- 这与传统插件安全不同，因为 skill 很可能同时携带：
  - 权限约定
  - 行为约束
  - 脚本
  - 持久化记忆
  - 工作流编排逻辑

### 官方强调的核心观念

- `MCP` 解决“模型怎么和工具说话”
- `AST10` 解决“这些工具最终被拿来做什么”

### 这部分可带出的核心观点

- AI 时代的“可执行模板”本身就是新的供应链和执行面
- 未来团队里共享的 prompt、workflow、skill、agent template，都可能需要像代码一样治理

## 3.6 变化五：OWASP MCP Top 10

### 已核实的官方信息

- OWASP 已上线 `MCP Top 10`
- 列表包括：
  - Token/Secret 暴露
  - Scope creep 导致的权限升级
  - Tool poisoning
  - 依赖投毒
  - 命令注入与执行
  - Intent flow subversion
  - 认证授权不足
  - 审计与遥测不足
  - Shadow MCP servers
  - Context injection 与过度共享

### 为什么值得讲

- 这说明 OWASP 已经把 AI 系统中的“协议层/连接层”视为正式安全边界
- 对技术团队来说，这非常像：
  - API 安全
  - 网关安全
  - 插件平台安全
  - 但它又叠加了 prompt、上下文、工具链和代理行为等新问题

### 对团队的启发

- 未来如果接入 MCP server，不是“装上就能用”
- 需要像接入第三方系统一样治理：
  - 权限
  - 来源
  - 数据边界
  - 审计
  - 沙箱

## 3.7 变化六：OWASP Non-Human Identities Top 10:2025

### 为什么这和 AI 时代强相关

- AI Agent、自动化工具、CI/CD、服务账户、本地插件、本地助手，都会大量使用非人身份：
  - API Key
  - Token
  - Service Account
  - 证书
  - 工作负载身份

### 官方列出的部分风险

- `Improper Offboarding`
- `Secret Leakage`
- `Vulnerable Third-Party NHI`
- `Insecure Authentication`
- `Overprivileged NHI`
- `Long-Lived Secrets`
- `Environment Isolation`

### 这部分在分享中可以怎么讲

- 以前我们主要担心“人被盗号”
- 现在更要担心“不是人，但权限很大、动作很多、管理很差的身份”
- 这和 AI 助手、自动化机器人、CI 账号、部署机器人高度相关

## 3.8 变化七：OWASP AI Exchange 与标准化/治理

### 已核实的官方信息

- `2025-05-06` 的 OWASP 博文提到：
  - AI Exchange 正在推动 AI 安全标准化
  - OWASP 通过官方标准合作，把开源安全实践带入标准制定

### 这说明什么

- OWASP 不只是在写“漏洞清单”
- 也在往“安全治理、标准对齐、监管落地”方向走

### 分享中可用的观点

- 如果说传统 OWASP 更像“开发者与渗透测试视角”
- 那 AI 时代的 OWASP 正在同时覆盖：
  - 工程视角
  - 平台视角
  - 治理视角
  - 标准视角

## 3.9 变化八：OWASP AIBOM 与 AI 供应链透明化

### 已核实的官方信息

- OWASP 已有 `AIBOM` 相关 initiative 与 generator

### 为什么重要

- AI 系统的供应链不只有代码依赖，还包括：
  - 模型
  - 数据集
  - embedding 模型
  - 推理服务
  - 第三方插件
  - agent skill

### 可讲的观点

- 传统软件世界有 `SBOM`
- AI 世界正在走向 `AIBOM`
- 这很适合和 `A03 Software Supply Chain Failures` 做呼应

## 3.10 变化九：OWASP 把深度伪造、AI 威胁情报、实战资源也纳入体系

### 已核实的官方信息

- 有 `Guide for Preparing and Responding to Deepfake Events`
- 有 `AI Threat Intelligence Initiative`
- 有 Agentic CTF、解决方案图谱等资源

### 这说明什么

- OWASP 在 AI 时代不是只给“漏洞名词解释”
- 而是在给：
  - 防守者指南
  - 事件响应思路
  - 演练素材
  - 能落地的实践资产

## 3.11 这一部分最适合提炼成的核心结论

1. OWASP 已经从“Web 漏洞 Top 10”扩展成“现代软件与 AI 系统安全体系”。
2. AI 安全不再只是模型问题，而是提示词、协议、技能、身份、供应链、治理的系统问题。
3. Agent、MCP、Skill、AIBOM、NHI 这些新名词，未来很可能会像今天的 JWT、SBOM、CSP 一样进入日常工程实践。

# 四、AI 时代如何增强 OWASP 在实际项目中的参与度和使用便捷性

## 4.1 你的想法：构建一个 OWASP agent skill

### 这个想法为什么好

- 它非常贴近团队真实研发流程
- 它不是“再写一份规范”，而是把规范变成可调用能力
- 它特别适合安全知识不均衡的团队

### 这个 skill 可以解决什么问题

- 开发在写方案时，快速做 OWASP 风险审视
- 开发在写代码后，请 AI 对照 OWASP 做初步审查
- Reviewer 在 CR 时，用统一口径补充安全检查
- 测试在出测试点时，让 AI 生成安全测试思路
- 运维在上线前，对配置和部署面做安全核对

## 4.2 OWASP agent skill 的初步设计想法

### 定位

- 一个“面向团队项目交付流程的 OWASP 审视助手”
- 不是替代安全专家，而是把安全规范前移、平民化、流程化

### 输入

- 需求文档
- 接口文档
- 方案设计
- 代码 diff
- 单个文件或模块代码
- 部署清单 / 配置片段
- 数据流说明

### 输出

- 命中的 OWASP 风险类别
- 风险原因解释
- 具体违反点
- 修复建议
- 推荐测试点
- 严重级别
- 置信度/需人工复核标记

### 可按四种模式设计

- `需求评审模式`
  - 判断是否存在越权、认证、敏感数据、异常流、审计缺口
- `方案评审模式`
  - 判断是否存在不安全设计、信任边界错误、权限模型缺陷
- `代码评审模式`
  - 判断是否存在注入、配置错误、日志泄露、文件上传风险等
- `上线检查模式`
  - 判断是否存在默认配置、暴露端点、弱密钥、缺日志等

### 技术实现思路

- 将 OWASP 安全规范拆成结构化规则库
- 每条规则至少包含：
  - 规则 ID
  - 适用场景
  - 风险说明
  - 触发信号
  - 修复建议
  - 映射的 OWASP 类别
- 将规则组织成适合 agent skill 调用的 Markdown/JSON 资产
- 根据输入类型选择不同检查模板

## 4.3 这个 skill 的价值

- 把“安全规范”从静态文档变成“研发随手可用能力”
- 降低大家对安全术语的门槛
- 让安全审查更早发生在需求、设计、编码阶段
- 让 code review 更一致，不再完全依赖 reviewer 个人经验

## 4.4 这个 skill 的边界和风险

- AI 不能代替正式安全审计
- 它可能：
  - 漏报
  - 误报
  - 不了解业务上下文
  - 不理解现网补偿控制
- 所以输出需要分层：
  - 明确问题
  - 可疑问题
  - 建议人工复核问题

## 4.5 比“单个 OWASP skill”更完整的一组方案

## 方案 A：OWASP 需求/方案评审助手

### 做什么

- 在需求评审、方案评审阶段自动生成安全问题清单

### 价值

- 解决 `A06 Insecure Design`
- 把安全前移到写代码之前

### 适合度

- 很适合做团队分享后的第一个试点

## 方案 B：OWASP 代码审查助手

### 做什么

- 在 PR/MR 阶段对 diff 进行 OWASP 映射审查

### 价值

- 让 code review 不只看可读性和功能正确性
- 可重点覆盖 `A01/A05/A07/A09/A10`

### 适合度

- 非常适合和你提出的 skill 结合

## 方案 C：OWASP 安全测试点生成助手

### 做什么

- 根据接口或模块自动生成安全测试点

### 价值

- 帮测试和开发快速补齐越权、注入、认证、文件上传、异常处理等测试场景

## 方案 D：OWASP 上线前基线检查助手

### 做什么

- 对配置、容器、网关、端口、证书、Header、日志策略做上线前检查

### 价值

- 直接对应 `A02/A03/A04/A08/A09`
- 对运维/平台很友好

## 方案 E：OWASP 安全问答 Copilot

### 做什么

- 开发随时提问：
  - “这个接口会不会有越权风险？”
  - “JWT 这样设计有问题吗？”
  - “上传 Excel 应该防什么？”

### 价值

- 持续培养安全意识
- 适合作为团队日常知识入口

## 方案 F：OWASP 安全知识库 + 可执行模板库

### 做什么

- 将团队常见场景沉淀成模板
  - 登录接口模板
  - 上传接口模板
  - 导出接口模板
  - 回调接口模板
  - 管理台接口模板

### 价值

- 用“安全 paved road”减少重复犯错
- 呼应 OWASP 对 secure design pattern / paved road 的强调

## 4.6 推荐的落地优先级

### 第一优先级：低成本快速见效

- `OWASP 方案评审助手`
- `OWASP 代码审查助手`
- `OWASP 安全问答 Copilot`

原因：

- 基本不需要深度改造现有系统
- 适合分享后立刻试点

### 第二优先级：工程化增强

- `OWASP 上线前基线检查助手`
- `OWASP 安全测试点生成助手`

原因：

- 需要接入更多流程和平台

### 第三优先级：体系化建设

- `OWASP 安全知识库 + 模板库`
- `OWASP agent skill 生态化`

原因：

- 价值大，但要长期维护

## 4.7 可供分享时抛出的“下一步行动项”

### 行动项建议 1

- 用 2 周时间做一个最小版 `OWASP code review skill`
- 先覆盖 8 到 12 条团队高频规则：
  - 越权
  - SQL 注入
  - XSS
  - 文件上传
  - JWT 生命周期
  - 敏感信息日志
  - 默认配置暴露
  - 密钥硬编码
  - 异常 fail-open

### 行动项建议 2

- 选 1 个典型业务模块试点
- 比较：
  - 没有 skill 时的 review 质量
  - 有 skill 后的 review 质量和效率

### 行动项建议 3

- 把 skill 输出结果沉淀为团队规范和 checklist
- 让“AI 提示”最终变成“团队制度资产”

## 4.8 这部分分享时可形成的结论

1. AI 不只是新的风险来源，也可以成为 OWASP 落地的放大器。
2. 最值得做的不是“让 AI 代替安全”，而是“让 AI 把安全能力嵌入日常研发动作”。
3. 一个可用的 OWASP agent skill，应该优先服务于需求评审、代码评审、测试点生成、上线检查这四个高频节点。

# 五、一些可直接用于演讲的创意表达

## 5.1 一个可能的开场问题

- “如果今天让大家各自写一份安全编码规范，多半能写出 SQL 注入、XSS、弱密码。但如果问：CI/CD 被投毒怎么办？Agent skill 被污染怎么办？AI 助手拿着过大的 token 到处跑怎么办？这时我们就会发现，OWASP 已经不是过去脑海里的那个样子了。”

## 5.2 一个核心主线

- `从“写代码别出漏洞”到“让整个研发与 AI 协作过程都更安全”`

## 5.3 一个适合全场反复回扣的句子

- “OWASP 的价值，不只是告诉我们哪里危险，而是帮助我们把‘知道危险’变成‘每天做事时顺手避开危险’。”

## 5.4 一个适合收尾的观点

- “如果规范永远躺在 PDF 里，它就只能靠记忆执行；如果规范进入 agent、进入 review、进入流水线，它才开始真正参与项目。”
