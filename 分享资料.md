# 一、开场：为什么今天还要专门讲 OWASP

## 1.1 我希望大家带走的不是“名词”，而是“默认动作”

今天这场分享，不是为了让大家记住更多漏洞名词，而是希望把几件事变成团队的默认动作：

- 写接口时，天然想到输入校验、权限校验、异常处理、日志审计。
- 写页面时，天然想到输出编码、富文本风险、文件上传风险。
- 做部署和运维时，天然想到默认配置、暴露面、日志、凭据、补丁。
- 在使用 AI 时，天然想到 prompt、tool、token、skill、供应链和权限边界。

如果安全规范永远只是 PDF，它只能靠记忆执行；如果它进入评审、代码、流水线和 AI 助手，它才开始真正参与项目。

## 1.2 为什么 OWASP 值得作为这次分享的主线

OWASP 的价值不只是它有名，而是它覆盖了我们最容易遇到的两类问题：

- 一类是“经典 Web 安全问题”，比如注入、越权、认证、日志、配置错误。
- 一类是“现代工程和 AI 带来的新问题”，比如软件供应链、非人身份、MCP、Agent、Skill。

换句话说，OWASP 已经不只是“教我们别写出 SQL 注入”，而是在帮助我们回答一个更大的问题：

`在现代研发方式里，怎样才能把安全真正嵌进日常工作。`

## 1.3 一句话概括今天的内容

今天我想讲的是：

`从“写代码别出漏洞”，走到“让整个研发和 AI 协作过程都更安全”。`

# 二、第一部分：从 `OWASP_zh.pdf` 看安全编码底线

## 2.1 这份中文 PDF 虽然老，但非常适合做“底座”

我们手头的 `OWASP_zh.pdf` 是《OWASP 安全编码规范快速参考指南》的中文版，时间比较早，但里面很多原则今天依然不过时。

它最重要的价值，不是告诉我们某个框架怎么配置，而是给出一组跨技术栈都成立的底线规则。它很适合当作团队安全编码规范的起点。

我建议把它理解成一句话：

`不管技术怎么变，Web 开发总有一些必须反复做到的安全常识。`

## 2.2 四个总原则

### 原则一：不要把客户端限制当成安全控制

前端校验、隐藏字段、菜单隐藏、按钮灰掉，都只能改善体验，不能替代安全控制。

对我们这种 Web 团队来说，这一点尤其重要。因为攻击者根本不会按页面走流程，他会直接改请求、改参数、改 Header、改 Cookie。

所以安全判断永远要问一句：

`如果前端完全失守，这个接口在服务端还能不能守住。`

### 原则二：安全控制要集中化、服务端化、可复用

OWASP 在很多章节里都强调“集中控制”。这对 Spring Boot 团队尤其关键。

我们最怕的不是“某个人不会写安全代码”，而是安全逻辑散在各个 Controller、各个项目、各个微服务里，最后全靠个人记忆。

更可靠的做法是把安全收口到统一能力里：

- 统一认证
- 统一授权
- 统一输入校验
- 统一异常处理
- 统一审计日志
- 统一安全基线

### 原则三：默认拒绝，而不是默认放行

很多安全事故不是因为“主动写错”，而是因为“默认就放过了”。

典型例子包括：

- 新增接口忘了加鉴权
- 新增对象存储桶默认公开
- 下游超时时，安全组件失效后走放行逻辑
- 错误场景用了一个危险的默认值

所以很多地方都应该 `deny by default`，特别是权限、配置、异常处理和文件访问。

### 原则四：安全问题通常是“链路没收口”

安全不只是代码问题，也不只是运维问题。

一条完整的风险链路，往往同时经过：

- 页面输入
- 服务端处理
- 数据库存储
- 日志记录
- 文件系统或对象存储
- 网关与部署配置

这也是为什么今天的分享要同时面对前端、后端、运维，而不是只讲某一种角色。

## 2.3 六个最值得团队重点记住的安全章节

为了控制 1 小时内的讲解深度，我建议这一部分不逐条念完整个 PDF，而是抓住最贴近我们日常开发的六个主题。

## 2.3.1 输入验证：所有输入都不可信

这是整份指南里最基础也最容易被低估的一条。

OWASP 原文里有几句很值得直接引用：

> “在可信系统（比如：服务器）上执行所有的数据验证。”
>
> “尽可能采用‘白名单’形式，验证所有的输入。”
>
> “在处理以前，验证所有来自客户端的数据，包括：所有参数、URL、HTTP 头信息……”

这里最想强调两点：

第一，输入不只是表单字段。对我们团队来说，下面这些都算输入：

- URL 参数
- JSON body
- Header
- Cookie
- 上传文件名
- 导入文件内容
- MQ 消息
- 第三方回调参数
- Redis 或数据库里再次读出来的数据

第二，校验不只是“非空”。

真正有效的输入校验至少要覆盖：

- 类型
- 长度
- 范围
- 枚举值
- 字符集
- 业务合法性

团队里最典型的错误场景有三个：

- 接口只校验字段非空，不校验状态流转是否合法。
- 排序字段或表名允许动态传入，最后被拼进 SQL。
- 文件下载接口接受路径参数，出现路径穿越。

如果这一部分只留一句给团队，我建议是：

`所有输入都不可信，服务端必须统一校验，而且要优先白名单。`

## 2.3.2 输出编码：安全不是“过滤脏字符”，而是“按上下文编码”

OWASP 原文里强调：

> “通过语义输出编码方式，对所有返回到客户端的来自于应用程序信任边界之外的数据进行编码。”

这里最容易被误解。很多人会以为只要把 `<` 和 `>` 替换掉就安全了，但真正的问题是：

`同一份数据，落在不同输出上下文里，风险完全不同。`

同样一段用户输入，如果出现在：

- HTML 文本节点
- HTML 属性
- JavaScript 片段
- URL 参数
- 响应头

它需要的处理方式都不一样。

这部分最适合拿前端和后端一起讲：

- 前端不要把框架默认转义当成万能保护。
- 后端不要用一次“统一转义”幻想一劳永逸。
- 富文本、Markdown、`v-html`、模板拼接、下载文件名回显，这些都要重点关注。

团队里很现实的例子是：

- 评论内容支持富文本，展示时没有白名单清洗。
- 报表页面把用户输入直接回显到 HTML 属性或 JS 片段中。
- 下载接口把文件名原样拼进响应头。

如果这一部分只留一句话：

`输出安全靠上下文编码，不靠一次性的全局替换。`

## 2.3.3 身份验证与会话管理：别把“登录成功”当成安全的终点

OWASP 在身份验证和会话管理上给了很多很朴素但很重要的要求，比如：

> “除了那些特定设为‘公开’的内容以外，对所有的网页和资源要求身份验证。”
>
> “身份验证的失败提示信息应当避免过于明确。”
>
> “注销功能应当完全终止相关的会话或连接。”
>
> “不要在 URL、错误信息或日志中暴露会话标识符。”

这部分我建议大家从“完整链路”理解，而不是只盯登录接口：

- 密码怎么存
- 登录失败怎么提示
- 是否有频控和锁定
- 是否支持 MFA
- Token 生命周期多长
- 登出后是否真的失效
- 切换账号后旧会话是否还有效

对我们团队来说，下面几类系统要特别警惕：

- 管理后台
- 运维台
- 配置中心
- 审批流系统
- 面向外部合作方的管理接口

今天很多系统都在用 JWT 或 SSO，但不要误以为“换了 token 方案就自动安全了”。

真正的问题仍然是：

- 能不能撤销
- 会不会被窃取
- 生命周期是否过长
- 风险登录是否有额外控制

如果这一部分只留一句话：

`身份验证不是只有登录那一刻，认证、会话、登出、找回密码、MFA 必须作为一个整体设计。`

## 2.3.4 访问控制：这是我们最容易真的出事故的一章

如果今天只让我选一个最值得团队反复讲的主题，我会优先选访问控制。

OWASP 原文里有几句非常关键：

> “只使用可信系统对象……以做出访问授权的决定。”
>
> “在每个请求中加强授权控制……”
>
> “限制只有授权的用户才能访问直接对象引用。”

我们在项目里最常见的权限问题，通常不是“没有登录”，而是：

- 只做了登录校验，没有做授权校验
- 只做了页面权限，没有做接口权限
- 只做了功能权限，没有做数据权限

这会直接演变成几类典型事故：

- 改一下 `userId`、`orderId`、`projectId` 就能看别人的数据
- 前端隐藏了按钮，但接口仍能直接调用
- 多租户系统没有真正按租户隔离数据
- 导出接口把跨部门数据一起导出来

这一部分最好直接用一句判断标准帮助大家记忆：

`权限控制至少要回答三个问题：你是谁、你能做什么、这份数据是不是你的。`

对于 Spring Boot 团队，我建议把权限控制放在服务端核心业务层，而不是只放在前端菜单或 Controller 表层。

## 2.3.5 错误处理与日志：对外要收敛，对内要完整

这部分是很多团队平时最容易忽略的。

OWASP 原文有两句话很适合直接引用：

> “不要在错误响应中泄露敏感信息……”
>
> “日志记录控制应当支持记录特定安全事件的成功或者失败操作。”

很多系统现在有日志，但未必有“安全日志”；有错误信息，但未必有“合适的错误信息”。

常见问题包括：

- 接口异常直接把堆栈返回给前端
- 日志打出 token、手机号、身份证、请求体
- 登录失败没有审计
- 权限失败没有审计
- 有日志平台，但没有告警规则和响应剧本

这一部分最重要的平衡是：

- 对外：错误信息尽量抽象，避免暴露实现细节
- 对内：关键安全事件必须可追踪、可审计、可告警

如果只留一句话：

`错误信息要收敛，安全日志要完整，而且日志里不能再制造新的泄露。`

## 2.3.6 系统配置、数据库与文件上传：很多事故根本不在业务代码里

团队谈安全时很容易只盯代码，但 OWASP 明确提醒我们，系统配置、数据库和文件管理本身就是高风险区。

这里我建议把三个主题合起来讲，因为它们有一个共同点：

`很多事故不是功能写错，而是环境、配置和资源边界没管住。`

适合引用的原文包括：

> “确保服务器、框架和系统部件安装了当前使用版本的所有补丁。”
>
> “使用强类型的参数化查询方法。”
>
> “连接字符串不应当在应用程序中硬编码……”
>
> “只验证文件类型扩展是不够的。”

对我们团队最现实的落点是：

- Spring Boot Actuator 暴露过多端点
- Swagger 或测试页面出现在生产
- Nginx/Tomcat 暴露版本信息
- 数据库账号权限过大
- MyBatis `${}` 动态拼接
- 上传目录可执行
- 对象存储桶默认公开
- 只看文件扩展名，不看文件头

这一部分我建议最后收束成一句：

`很多高风险功能，不在“业务逻辑的 happy path”里，而在配置、依赖、文件、数据库这些基础设施边角。`

## 2.4 第一部分的小结

如果把第一部分压缩成十条团队底线，我建议是：

1. 所有输入都不可信，服务端统一校验。
2. 所有输出都按上下文编码。
3. 认证、会话、登出、找回密码要按完整链路设计。
4. 授权必须做到接口层和数据层，而不是只做页面层。
5. 任何 SQL、命令、模板、表达式拼接都要高度警惕。
6. 敏感数据、密钥、证书、连接串不能明文散落。
7. 错误对外收敛，日志对内完整。
8. 文件上传、下载、导入、导出都要当高风险能力设计。
9. 默认配置不可信，安全基线要自动化。
10. 安全不是上线前扫一次，而是需求到部署全过程。

# 三、第二部分：`OWASP Top 10:2025` 与我们团队最相关的风险

## 3.1 一个先说明的事实

截至 `2026-05-26`，OWASP 官方已经发布 `OWASP Top 10:2025`。这意味着我们今天不能再只用 2021 的框架来理解当前的主流风险视角。

2025 版 Top 10 排名如下：

1. `A01 Broken Access Control`
2. `A02 Security Misconfiguration`
3. `A03 Software Supply Chain Failures`
4. `A04 Cryptographic Failures`
5. `A05 Injection`
6. `A06 Insecure Design`
7. `A07 Authentication Failures`
8. `A08 Software or Data Integrity Failures`
9. `A09 Security Logging & Alerting Failures`
10. `A10 Mishandling of Exceptional Conditions`

## 3.2 2025 版最值得强调的三个变化

### 变化一：安全配置错误升得很高

`A02 Security Misconfiguration` 排到第 2 位，说明今天大量真实问题来自配置、暴露面和缺少加固，而不是只有编码错误。

### 变化二：软件供应链成为核心风险

`A03 Software Supply Chain Failures` 排到第 3 位，说明现代应用的安全边界已经远不止“我们自己写的代码”。

### 变化三：异常条件处理不当进入 Top 10

`A10 Mishandling of Exceptional Conditions` 的出现很有代表性。它提醒我们：失败路径本身就是攻击面，很多事故发生在超时、空值、回退、异常兜底这些地方。

## 3.3 对我们团队最值得详细讲的六项

如果要在 1 小时里兼顾完整度和重点，我建议详细讲六项，剩余四项做短讲。

## 3.3.1 A01 Broken Access Control

这是最应该排第一位讲给团队听的风险。

它的核心不是“系统有没有登录”，而是：

- 用户能不能访问不属于自己的资源
- 用户能不能执行不该执行的操作
- 系统有没有把对象级、字段级、租户级边界守住

在我们团队里，最常见的表现是：

- 改 ID 看别人数据
- 菜单隐藏了，但接口仍能调
- 审批、导出、下载、报表接口漏了数据权限
- 多租户隔离只做在页面上，没有做在服务端查询上

防范这类风险，最关键的不是“多加几个 if”，而是建立统一授权模型：

- 统一身份
- 统一角色/权限模型
- 统一资源归属判断
- 关键对象的 ownership 校验
- 越权专项测试

一句话总结：

`越权问题本质上是“业务边界没有被系统真正执行”。`

## 3.3.2 A02 Security Misconfiguration

这类问题特别适合讲给运维和后端，也必须让前端知道它为什么重要。

它常见于：

- 默认配置没关
- 不必要的服务、页面、端点对外暴露
- 错误页暴露细节
- 安全头缺失
- 云资源、对象存储、容器、网关没有做收口

对我们团队最现实的例子：

- Spring Boot Actuator 暴露太多端点
- Swagger 或测试接口跑到了生产
- Nginx/Tomcat 暴露版本信息
- 对象存储桶或文件服务默认公开

这类问题最麻烦的地方是，它常常不是某个开发同学“写错了一行代码”，而是整个交付链路缺少统一安全基线。

所以它的治理方式必须偏工程化：

- 基线模板
- 自动检查
- 环境差异收敛
- 暴露面盘点
- 上线前配置核查

## 3.3.3 A03 Software Supply Chain Failures

这项排名升到第 3，非常值得团队重视。

过去大家常说“不要引入有漏洞的依赖”，但今天的供应链风险已经远不止依赖版本：

- Maven 传递依赖
- 镜像来源
- 制品仓库
- CI/CD 权限
- 发布流程
- 插件、脚本、自动化工具

对我们团队来说，最现实的几个问题是：

- 依赖长期无人盘点
- 开发机、CI、制品库权限过大
- 生产包不是同一份构建产物
- 发布链路没有完整留痕

这一项和传统“安全编码”最大的区别在于：

`哪怕业务代码写得很好，构建和交付链路被污染，结果仍然可能是不安全的。`

所以比较关键的治理动作包括：

- 做依赖和镜像清单
- 关注高危漏洞公告
- 构建产物可追溯
- 对关键仓库和流水线做最小权限与 MFA
- 逐步引入 SBOM、签名和来源证明

## 3.3.4 A05 Injection

虽然注入在 2025 版不再排第 1，但它仍然是最容易被理解、也最容易真的出事的一类风险。

对我们团队来说，注入绝不只等于 SQL 注入，还包括：

- SQL / HQL / JPQL 注入
- 命令注入
- 表达式注入
- 模板注入
- 未来 LLM 场景下的 Prompt Injection

最常见的错误心智是：

`我们平时都用 ORM，所以应该不会有注入了。`

但实际风险仍然很多：

- MyBatis `${}` 动态拼接
- 排序字段、表名、过滤表达式直接传入
- shell 命令拼接
- 表达式引擎或脚本执行能力暴露

防范注入问题有一个很稳定的思路：

- 能不用解释器式接口就不用
- 能结构化传参就不要字符串拼接
- 必要时用白名单约束可变部分
- 对动态能力做专门审计

一句话总结：

`注入的本质，是把“不可信输入”当成“程序指令的一部分”。`

## 3.3.5 A06 Insecure Design

这项风险很适合拿来提醒团队：安全不只是实现问题，更是设计问题。

它关注的是：

- 安全控制有没有被设计出来
- 业务流程有没有滥用路径
- 系统是不是只为“正常用户”设计，而没有为“恶意行为”设计

现实中，这类问题经常体现在：

- 密码找回逻辑过弱
- 审批流或退款流可以被绕过
- 多租户隔离先天就薄弱
- 抢购、领券、接口批量调用没有反滥用设计

这一项特别值得带给团队一个观念：

`很多严重问题不是因为代码写错，而是因为设计阶段根本没把攻击者放进场景里。`

所以在需求评审和方案评审阶段，就应该开始问：

- 如果有人恶意使用这个流程，会怎样？
- 如果有人绕开前端直接打接口，会怎样？
- 如果鉴权组件失败了，会怎样？

## 3.3.6 A10 Mishandling of Exceptional Conditions

这是 2025 版新增的类别，也是我特别建议重点讲的一项。

很多系统平时测试的是“正常流程”，但真实事故更容易出在：

- 参数缺失
- 空值
- 超时
- 重试
- 权限组件失败
- 网络异常
- 并发竞争
- 回滚不完整

为什么这项值得强调？

因为今天我们的系统越来越依赖：

- 微服务
- 网关
- 缓存
- MQ
- 外部 API
- AI 服务

一旦某个环节失败，如果系统用的是“先放过、先降级、先返回默认值”这种思路，就可能直接把安全控制绕开。

这一项我最建议大家记住一句话：

`关键安全能力在异常情况下必须 fail-secure，而不是 fail-open。`

## 3.4 其余四项的短讲版本

## A04 Cryptographic Failures

重点不是“有没有加密”，而是：

- 算法是否过时
- 密钥是否硬编码
- 是否用错场景
- 是否真正保护了敏感数据和传输链路

## A07 Authentication Failures

重点不是“能不能登录”，而是：

- 是否抗弱密码、撞库、暴力破解
- 找回密码是否过弱
- 会话/token 生命周期是否合理
- 管理系统是否启用 MFA

## A08 Software or Data Integrity Failures

重点是：

- 我们是否信任了不该信任的代码、制品、数据或更新渠道
- 是否验证来源和完整性

## A09 Security Logging & Alerting Failures

重点是：

- 有没有把关键安全事件记录下来
- 能不能及时发现异常
- 日志是不是会泄露更多问题

## 3.5 第二部分的小结

如果把 Top 10:2025 讲给我们团队，我最希望大家记住的不是排名，而是三层变化：

- 第一层，经典风险仍然存在，尤其是越权、注入、认证。
- 第二层，工程化风险越来越重要，尤其是配置、供应链、完整性、日志。
- 第三层，失败路径已经正式成为攻击面，异常处理不能再只当稳定性问题看。

# 四、第三部分：AI 时代，OWASP 发生了什么变化

## 4.1 一个总体判断

OWASP 正在从“经典 Web 安全知识库”扩展成“现代软件与 AI 系统安全体系”。

这不是多加了几个 AI 术语，而是 OWASP 的关注范围真的变了：

- 从 Web 应用扩展到 LLM、Agent、MCP、Skill
- 从编码缺陷扩展到协议、身份、供应链、治理
- 从指南扩展到实战资源、清单、标准化和生态项目

## 4.2 最值得团队知道的几个变化

## 4.2.1 OWASP Top 10 for LLM 已升级为 GenAI Security Project

这说明 OWASP 不再把 AI 安全理解成“给 LLM 列 10 条风险”这么简单，而是开始建设一整套持续治理体系。

这对团队的启发是：

`只要我们开始接入大模型，无论是做问答、RAG、代码助手、文档处理还是 Agent，我们就已经进入一个新的安全面。`

## 4.2.2 LLM Top 10:2025 说明 AI 风险不是只有“幻觉”

2025 版里面至少有几项非常值得团队注意：

- Prompt Injection
- Sensitive Information Disclosure
- Supply Chain
- Improper Output Handling
- Excessive Agency
- Vector and Embedding Weaknesses
- Unbounded Consumption

这说明 AI 安全的重点已经落在：

- 输入会不会污染模型
- 输出会不会直接影响系统
- Agent 权限会不会过大
- 成本和资源会不会失控

## 4.2.3 MCP Top 10 和 Agentic Skills Top 10 很值得重点讲

这是最能体现“OWASP 与时俱进”的两个例子。

为什么？

因为它们抓到的是 AI 时代真正新的攻击面：

- MCP 关注模型如何连接工具和外部系统
- Skill 关注 AI 被教会“怎么做事”的过程本身

如果换成传统研发语言，可以把它理解为：

- MCP 像“AI 系统的接口协议层”
- Skill 像“AI 系统的可执行流程模板层”

这两个层面一旦出问题，影响的已经不是一段文本，而是：

- token
- 文件
- 命令
- 外部系统调用
- 自动化执行结果

## 4.2.4 Non-Human Identities Top 10 也非常适合我们团队

AI 时代会出现越来越多“不是人，但权限很大”的身份：

- API Key
- Service Account
- CI 账号
- 部署机器人
- Agent token
- MCP server 凭据

这类身份如果管理不好，风险往往比普通用户账号更大，因为它们：

- 权限广
- 生命周期长
- 使用频繁
- 监控薄弱

## 4.2.5 AIBOM、AI Exchange、Deepfake 指南说明 OWASP 已走向治理层

OWASP 已经不只是在告诉开发者“怎么写更安全”，也在推动：

- AI 供应链透明化
- AI 安全标准化
- AI 事件响应准备
- AI 威胁情报和实战演练

这一部分我建议用一句话收束：

`AI 时代的 OWASP，正在同时覆盖开发、平台、安全治理和标准对齐。`

## 4.3 第三部分的小结

如果要把这一部分讲得不空泛，我建议给团队一个非常明确的结论：

`AI 不只是新工具，它正在引入新的输入面、执行面、身份面和供应链面。`

所以 OWASP 的演进，其实正是在回应这个现实。

# 五、第四部分：如何让 AI 增强 OWASP 在项目中的参与度和使用便捷性

## 5.1 核心判断

AI 不应该替代安全专家，但非常适合做两件事：

- 把安全能力前移
- 把安全规范嵌入日常研发动作

也就是说，最值得做的不是“让 AI 帮我们扫所有漏洞”，而是：

`让 AI 在需求评审、方案评审、代码评审、测试点设计、上线检查这些高频动作里，顺手把 OWASP 带进来。`

## 5.2 你提出的 `OWASP agent skill`，是一个很好的切入点

我很认可这个方向，因为它有三个优势：

- 它贴近真实研发流程
- 它把静态规范变成可调用能力
- 它对团队成员的安全水平要求更友好

这个 skill 可以服务至少四种场景：

- 需求评审：看有没有越权、认证、数据保护、审计缺口
- 方案评审：看有没有不安全设计、信任边界错误、权限模型缺陷
- 代码评审：看有没有注入、日志泄露、文件上传、配置错误
- 上线检查：看有没有默认配置、暴露端点、弱凭据、缺少日志

## 5.3 一个更完整、也更容易落地的方案组合

我建议不要把它只理解成“做一个 skill”，而是理解成“一组让 OWASP 进入研发流程的 AI 能力”。

## 5.3.1 第一优先级：快速试点

### A. OWASP 方案评审助手

输入方案文档、接口文档、流程图，让 AI 从 OWASP 视角产出问题清单。

这一步最能前移 `A06 Insecure Design`。

### B. OWASP 代码审查助手

输入代码 diff 或模块代码，让 AI 对照 OWASP 给出风险点、原因和修复建议。

这一步最适合覆盖：

- 越权
- 注入
- 认证/会话
- 敏感信息日志
- 文件上传
- 异常 fail-open

### C. OWASP 安全问答 Copilot

让开发在日常工作中直接提问：

- “这个接口会不会有越权风险？”
- “JWT 这样设计是否合理？”
- “上传 Excel 应该防什么？”

这一类能力的价值在于持续培养安全意识，而不是只在上线前集中补课。

## 5.3.2 第二优先级：工程化增强

### D. OWASP 安全测试点生成助手

根据接口和业务流程，自动生成越权、注入、认证、文件上传、异常处理等测试点。

### E. OWASP 上线前基线检查助手

对配置、镜像、端点、证书、Header、日志和凭据做核查，重点覆盖 `A02/A03/A04/A08/A09`。

## 5.3.3 第三优先级：长期建设

### F. OWASP 知识库与模板库

把团队高频能力沉淀成安全模板：

- 登录接口模板
- 上传接口模板
- 导出接口模板
- 回调接口模板
- 管理台接口模板

这其实是在打造一条“安全的默认路径”。

## 5.4 `OWASP agent skill` 最小可行版本建议

如果分享之后要落一个试点，我建议先做一个最小版，不要一开始就追求覆盖全部 OWASP。

第一版可以先覆盖 8 到 12 条团队高频规则：

- 越权
- SQL 注入
- XSS / 输出编码
- 文件上传
- JWT 生命周期
- 敏感信息日志
- 密钥硬编码
- 默认配置暴露
- 异常 fail-open

skill 的输出建议固定成下面几类：

- 命中的 OWASP 类别
- 风险原因
- 可能的攻击方式
- 修复建议
- 推荐测试点
- 严重级别
- 是否需要人工复核

## 5.5 这一部分最想推动的三个行动项

### 行动项 1

选一个典型项目模块，试做 `OWASP code review skill`。

### 行动项 2

把 skill 的输出结果和团队现有 CR 习惯做对比，验证是否能提升 review 质量和一致性。

### 行动项 3

把经过验证的规则沉淀成团队自己的安全规范和模板库，让 AI 输出最终反哺制度资产。

## 5.6 第四部分的小结

我最希望这部分传达出去的一句话是：

`AI 最有价值的地方，不是替我们“负责安全”，而是让安全更早、更频繁、更低门槛地出现在每一次研发动作里。`

# 六、收尾：今天最值得记住的四句话

## 6.1 四句总结

1. `OWASP_zh.pdf` 虽然老，但它提供的是今天仍然有效的安全底线。
2. `OWASP Top 10:2025` 告诉我们，现代安全风险已经从“代码漏洞”扩展到“配置、供应链、异常处理和完整性”。

4. 最值得做的不是再多写一份规范，而是把 OWASP 通过 AI 和工程化能力嵌进日常研发流程。

## 6.2 一个适合结束时说的句子

`如果规范永远躺在文档里，它只能靠记忆执行；如果规范进入评审、进入代码、进入流水线、进入 agent，它才开始真正改变项目。`

# 七、附：讲述提示

## 7.1 讲这一场时的节奏建议

- 第一部分不要念清单，要讲“为什么这些规则今天还有效”。
- 第二部分不要平均用力，要重点讲 `A01/A02/A03/A05/A06/A10`。
- 第三部分不要堆新名词，要始终回答“这为什么和我们团队有关”。
- 第四部分不要讲成科幻，要给出能在团队里试点的最小动作。

## 7.2 哪些地方适合讲真实例子

- 越权：改一个 ID 就能看到别人的数据
- 注入：排序字段拼进 SQL
- 配置错误：测试端点出现在生产
- 文件上传：只看扩展名，不看文件头
- 异常处理：下游失败后默认放行
- AI 场景：给 agent 太大的 token 和工具权限

## 7.3 哪些地方可以和听众互动

- “你们见过最常见的越权问题是什么？”
- “大家所在项目里，生产环境有没有还开着 Swagger 或测试端点？”
- “如果以后项目接 LLM，最担心的是 prompt、token、工具，还是数据泄露？”
- “如果让团队下周就试点一个 AI 安全能力，最值得先做哪一个？”
