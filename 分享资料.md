# 一、开场：为什么今天还要专门讲 OWASP

## 1.1 我希望大家带走的不是“名词”，而是“默认动作”

今天这场分享，不是为了让大家记住更多漏洞名词，而是希望把几件事变成团队的默认动作：

- 写接口时，天然想到输入校验、权限校验、异常处理、日志审计。
- 写页面时，天然想到输出编码、富文本风险、文件上传风险。
- 做部署和运维时，天然想到默认配置、暴露面、日志、凭据、补丁。
- 在使用 AI 时，天然想到 prompt、tool、token、skill、供应链和权限边界。

如果安全规范永远只是 PDF，它只能靠记忆执行；如果它进入评审、代码、流水线和 AI 助手，它才开始真正参与项目。

## 1.2 为什么 OWASP 值得作为这次分享的主线

OWASP 的价值不只是它有名，而是它覆盖了我们最容易遇到的两类问题：

- 一类是“经典 Web 安全问题”，比如注入、越权、认证、日志、配置错误。
- 一类是“现代工程和 AI 带来的新问题”，比如软件供应链、非人身份、MCP、Agent、Skill。

换句话说，OWASP 已经不只是“教我们别写出 SQL 注入”，而是在帮助我们回答一个更大的问题：

`在现代研发方式里，怎样才能把安全真正嵌进日常工作。`

## 1.3 一句话概括今天的内容

今天我想讲的是：

`从“写代码别出漏洞”，走到“让整个研发和 AI 协作过程都更安全”。`

# 二、第一部分：从 `OWASP_zh.pdf` 看安全编码底线

## 2.1 这份中文 PDF 虽然老，但非常适合做“底座”

我们手头的 `OWASP_zh.pdf` 是《OWASP 安全编码规范快速参考指南》的中文版，时间比较早，但里面很多原则今天依然不过时。

它最重要的价值，不是告诉我们某个框架怎么配置，而是给出一组跨技术栈都成立的底线规则。它很适合当作团队安全编码规范的起点。

我建议把它理解成一句话：

`不管技术怎么变，Web 开发总有一些必须反复做到的安全常识。`

## 2.2 四个总原则

### 原则一：不要把客户端限制当成安全控制

前端校验、隐藏字段、菜单隐藏、按钮灰掉，都只能改善体验，不能替代安全控制。

对我们这种 Web 团队来说，这一点尤其重要。因为攻击者根本不会按页面走流程，他会直接改请求、改参数、改 Header、改 Cookie。

所以安全判断永远要问一句：

`如果前端完全失守，这个接口在服务端还能不能守住。`

### 原则二：安全控制要集中化、服务端化、可复用

OWASP 在很多章节里都强调“集中控制”。这对 Spring Boot 团队尤其关键。

我们最怕的不是“某个人不会写安全代码”，而是安全逻辑散在各个 Controller、各个项目、各个微服务里，最后全靠个人记忆。

更可靠的做法是把安全收口到统一能力里：

- 统一认证
- 统一授权
- 统一输入校验
- 统一异常处理
- 统一审计日志
- 统一安全基线

### 原则三：默认拒绝，而不是默认放行

很多安全事故不是因为“主动写错”，而是因为“默认就放过了”。

典型例子包括：

- 新增接口忘了加鉴权
- 新增对象存储桶默认公开
- 下游超时时，安全组件失效后走放行逻辑
- 错误场景用了一个危险的默认值

所以很多地方都应该 `deny by default`，特别是权限、配置、异常处理和文件访问。

### 原则四：安全问题通常是“链路没收口”

安全不只是代码问题，也不只是运维问题。

一条完整的风险链路，往往同时经过：

- 页面输入
- 服务端处理
- 数据库存储
- 日志记录
- 文件系统或对象存储
- 网关与部署配置

这也是为什么今天的分享要同时面对前端、后端、运维，而不是只讲某一种角色。

## 2.3 这一部分最好按 PDF 原始父项讲，但讲解要有轻重

你提到的两个问题本质上都指向同一件事：这一部分不能只讲“我挑出来的几个主题”，而要尽量保留 `OWASP_zh.pdf` 的原始结构。

所以更合适的讲法是：

- 不再把 `身份验证和密码管理` 与 `会话管理` 合并。
- 不再遗漏 `通讯安全`、`数据保护` 这类看上去“像运维”但其实开发同样要负责的父项。
- 每个父项都单独讲，但讲解深浅分层。

如果按分享时间做取舍，我建议分成两层：

- 重点展开：输入验证、输出编码、身份验证和密码管理、会话管理、访问控制、错误处理和日志。
- 必须覆盖：加密规范、数据保护、通讯安全、系统配置、数据库安全、文件管理、通用编码规范。

这样既尊重原 PDF 的结构，也能把更多时间留给最核心、最常见、最容易被忽略的风险。

## 2.3.1 输入验证：所有输入都不可信

这是整份指南里最基础、也最容易被团队“以为自己已经做了”的一条。

### PDF 核心检查项原文

> “在可信系统（比如：服务器）上执行所有的数据验证。”
>
> “应当为应用程序应提供一个集中的输入验证规则。”
>
> “尽可能采用‘白名单’形式，验证所有的输入。”
>
> “在处理以前，验证所有来自客户端的数据，包括：所有参数、URL、HTTP 头信息（比如：cookie 名字和数据值）。”

这一组原文背后最需要强调两件事。

第一，输入不只是表单字段。对我们团队来说，下面这些都算输入：

- URL 参数
- JSON body
- Header
- Cookie
- 上传文件名
- 导入文件内容
- MQ 消息
- 第三方回调参数
- Redis 或数据库里再次读出来的数据

第二，校验不只是“非空”。真正有效的输入校验至少要覆盖：

- 类型
- 长度
- 范围
- 枚举值
- 字符集
- 规范化后的格式
- 业务合法性

团队里最典型的错误场景有三个：

- 接口只校验字段非空，不校验状态流转是否合法。
- 排序字段、表名、列名允许动态传入，最后被拼进 SQL。
- 文件下载或预览接口接受路径参数，最后出现路径穿越。

如果这一部分只留一句给团队，我建议是：

`所有输入都不可信，服务端必须统一校验，而且要优先白名单。`

## 2.3.2 输出编码：安全不是“过滤脏字符”，而是“按上下文编码”

这部分在前后端协作里最容易被误解。很多人以为把 `<`、`>` 之类字符替换掉就够了，但 OWASP 真正强调的是“语义化、按上下文编码”。

### PDF 核心检查项原文

> “在可信系统（比如：服务器）上执行所有的编码。”
>
> “为每一种输出编码方法采用一个标准的、已通过测试的规则。”
>
> “通过语义输出编码方式，对所有返回到客户端的来自于应用程序信任边界之外的数据进行编码。”
>
> “对于操作系统命令，净化所有不可信数据输出。”

同一份用户输入，如果出现在下面这些位置：

- HTML 文本节点
- HTML 属性
- JavaScript 片段
- URL 参数
- 响应头

它需要的处理方式都不一样。

所以这一章最值得团队记住的不是“做转义”，而是：

- 前端不要把框架默认转义当成万能保护。
- 后端不要幻想“一次统一转义”可以覆盖所有场景。
- 富文本、Markdown、`v-html`、模板拼接、文件名回显、跳转 URL，都属于高风险输出点。

最适合拿来讲的真实例子包括：

- 评论或公告支持富文本，展示时没有白名单清洗。
- 报表页面把用户输入直接回显到 HTML 属性或 JS 片段。
- 下载接口把文件名原样拼进响应头，导致响应头注入或 XSS。

如果这一部分只留一句话：

`输出安全靠上下文编码，不靠一次性的全局替换。`

## 2.3.3 身份验证和密码管理：别把“能登录”当成“认证没问题”

这一章要单独讲，因为它关注的是“你怎么确认一个人是谁”，而不是“登录后那段会话怎么维持”。

### PDF 核心检查项原文

> “除了那些特定设为‘公开’的内容以外，对所有的网页和资源要求身份验证。”
>
> “在任何可能的情况下，建立并使用标准的、已通过测试的身份验证服务。”
>
> “如果您的应用程序管理着凭证的存储，那么应当保证只保存了通过使用强加密单向 salted 哈希算法得到的密码，并且只有应用程序具有对保存密码和密钥的表/文件的写权限（如果可以避免的话，不要使用 MD5 算法）。”
>
> “为高度敏感或重要的交易账户使用多因子身份验证机制。”

OWASP 这章的核心要求包括：

- 除公开资源外，默认要求身份验证。
- 优先使用集中式、成熟的认证服务，而不是各系统各写一套。
- 密码必须做强哈希存储，不能明文、不能可逆、不要再碰 MD5。
- 登录失败提示不要泄露“账号存在/不存在”之类过细信息。
- 对敏感账号和高价值操作，要考虑锁定、频控、多因子、找回密码保护。

这部分对我们团队最值得展开的，不是“密码复杂度到底几位”这种细枝末节，而是几个高频误区：

- 只做登录接口，不把找回密码、短信验证码、邮箱链接、初始密码发放纳入认证链路。
- 把内部系统当作“熟人系统”，弱化复杂度、锁定、MFA、异地风险识别。
- 以为接入 SSO 或 OAuth2 之后，认证问题就自动消失了。

管理后台、运维台、审批台、配置中心、面向合作方的接口门户，都应该被当成认证高敏区。

如果这一部分只留一句话：

`身份验证的重点不是“能不能登进去”，而是凭证如何存、失败如何控、找回如何防、敏感场景如何加固。`

## 2.3.4 会话管理：登录成功以后，风险才刚刚开始

这章必须和上面分开讲，因为很多系统认证没出事，真正出事的是会话。

### PDF 核心检查项原文

> “会话标识符必须总是在一个可信系统（比如：服务器）上创建。”
>
> “如果一个会话在登录以前就建立，在成功登录以后，关闭该会话并创建一个新的会话。”
>
> “注销功能应当完全终止相关的会话或连接。”
>
> “不要在 URL、错误信息或日志中暴露会话标识符。会话标识符应当只出现在 HTTP cookie 头信息中。”

OWASP 在这部分强调的点非常朴素，但几乎条条都常见：

- 会话 ID 由服务端生成，而不是客户端自带或可预测。
- 登录成功后要切换成新会话，避免会话固定攻击。
- 注销必须真正让会话失效，而不是只跳回登录页。
- 不要在 URL、错误信息、日志中暴露会话标识。
- Cookie 要正确设置 `Secure`、`HttpOnly` 等属性。

今天很多系统不再用传统 `Session`，而是用 `JWT`、`SSO`、网关令牌，但会话问题并没有消失，只是换了表现形式：

- Token 能不能撤销
- 生命周期是不是过长
- 切换账号后旧 token 是否仍有效
- 浏览器、本地缓存、日志里有没有把 token 留下来
- 是否能把高风险操作和低风险浏览共用同一份长期凭证

最适合对团队反复强调的一句是：

`认证解决“你是谁”，会话管理解决“你接下来这一路还能不能持续被安全地识别”。`

## 2.3.5 访问控制：这是最容易真正出事故的一章

如果今天只让我选一个最值得团队反复讲的父项，我会优先选访问控制。

### PDF 核心检查项原文

> “只使用可信系统对象（比如：服务器端会话对象）以做出访问授权的决定。”
>
> “使用一个单独的全站点部件以检查访问授权。”
>
> “在每个请求中加强授权控制，包括：服务器端脚本产生的请求，‘includes’和来自象 AJAX 和 FLASH 那样的富客户端技术的请求。”
>
> “限制只有授权的用户才能访问直接对象引用。”

OWASP 这章的关键词很清楚：

- 每个请求都要做授权控制。
- 不能只做页面权限，还要做接口权限和数据权限。
- 不能相信客户端传来的角色、部门、租户、对象归属。
- 直接对象引用必须受控，不能改一个 ID 就越权。

我们在项目里最常见的权限问题，往往不是“没有登录”，而是：

- 只做了登录校验，没有做授权校验。
- 只做了功能权限，没有做数据权限。
- 前端隐藏了按钮，但接口仍能直接调用。
- 多租户系统只在页面上区分租户，没有在查询条件和服务层真正收口。

这会直接演变成几类典型事故：

- 改一下 `userId`、`orderId`、`projectId` 就能看别人的数据。
- 导出接口一下子把跨部门或跨租户数据全带走。
- “仅管理员可见”的页面虽然藏起来了，但接口没拦住。

这一部分最好直接留一句判断标准帮助大家记忆：

`权限控制至少要回答三个问题：你是谁、你能做什么、这份数据是不是你的。`

## 2.3.6 加密规范：别把 Base64、摘要、哈希、加密混成一件事

这一章在很多团队里容易被讲得太虚，或者只剩一句“上 HTTPS 就行”。其实它讲的是更底层的安全能力：算法、密钥、随机数、密码存储、证书使用。

### PDF 核心检查项原文

> “所有用于保护来自应用程序用户秘密信息的加密功能都必须在一个可信系统（比如：服务器）上执行。”
>
> “安全的处理加密模块失败的操作。”
>
> “为防范对随机数据的猜测攻击，应当使用加密模块中已验证的随机数生成器生成所有的随机数、随机文件名、随机 GUID 和随机字符串。”
>
> “建立并使用相关的政策和流程以实现加、解密的密钥管理。”

对分享来说，最值得抓住的是几个常见误区：

- 把 Base64、可逆加密、单向哈希混为一谈。
- 密码仍然用弱哈希，或者把“加盐”做成形式主义。
- 自己拼加密方案，自己定密钥格式，自己造 token 签名逻辑。
- 密钥、证书、连接串散落在配置文件、脚本、群文件、仓库里。

对团队最相关的落地理解可以收敛成四点：

- 密码存储和敏感数据加密是两类问题，不要用同一种手法处理。
- 算法和模式优先用成熟库、成熟默认值，不要自造。
- 密钥管理和代码管理必须分开，密钥生命周期要可轮换、可吊销。
- 不要把“能解出来”当成成功标准，要把“密钥是否守住”当成成功标准。

如果这一部分只留一句话：

`真正危险的不是“不会加密”，而是“以为自己已经做了加密”。`

## 2.3.7 错误处理和日志：对外要收敛，对内要完整

这部分是很多团队平时最容易忽略的。

### PDF 核心检查项原文

> “不要在错误响应中泄露敏感信息，包括：系统的详细信息、会话标识符或者帐号信息。”
>
> “使用通用的错误消息并使用定制的错误页面。”
>
> “日志记录控制应当支持记录特定安全事件的成功或者失败操作。”
>
> “记录所有失败的输入验证。记录所有的身份验证尝试，特别是失败的验证。记录所有失败的访问控制。”

OWASP 在这里强调的不是“多打日志”，而是：

- 不要在错误响应里暴露堆栈、账号、路径、表名、会话标识、内部实现细节。
- 关键安全事件要有审计记录，尤其是失败的输入验证、失败的认证、失败的授权。
- 日志不能反过来成为新的泄露面。

很多系统现在有日志，但未必有“安全日志”；有错误信息，但未必有“合适的错误信息”。

常见问题包括：

- 接口异常直接把堆栈返回给前端。
- 日志打出 token、手机号、身份证、SQL、请求体。
- 登录失败没有审计，权限失败没有审计。
- 虽然有日志平台，但没有告警规则和响应剧本。

这一部分最重要的平衡是：

- 对外：错误信息尽量抽象，避免暴露实现细节。
- 对内：关键安全事件必须可追踪、可审计、可告警。

如果只留一句话：

`错误信息要收敛，安全日志要完整，而且日志里不能再制造新的泄露。`

## 2.3.8 数据保护：很多泄露不是“被黑进去”，而是“自己放出来了”

这章经常被低估，因为它不像注入、越权那样“像漏洞”，但很多真实事故恰恰发生在这里。

### PDF 核心检查项原文

> “授予最低权限，以限制用户只能访问为完成任务所需要的功能、数据和系统信息。”
>
> “即使在服务器端，任然要加密存储的高度机密信息，比如，身份验证的验证数据。”
>
> “不要在 HTTP GET 请求参数中包含敏感信息。”
>
> “禁止客户端缓存网页，因为可能包含敏感信息。‘Cache-Control: no-store’，可以和 HTTP 报头控制‘Pragma: no-cache’一起使用。”

这部分最值得团队记住的是：

- 敏感数据不要明文散落在数据库、缓存、临时文件、对象存储里。
- 不要在 GET 参数里放 token、手机号、工号、重置口令之类敏感信息。
- 敏感页面和敏感响应要控制缓存，不要默认允许浏览器和代理留痕。
- 测试数据、导出文件、备份文件也属于数据保护范围，不是“业务之外的事”。

特别适合讲的团队场景有：

- URL 里直接带登录票据、手机号、身份证号。
- 导出文件长期挂在公开 URL。
- 测试库从生产拷贝数据，但脱敏没有真正落实。
- 敏感 Excel、CSV、PDF 先落地到临时目录，再被长期遗留。

如果这一部分只留一句话：

`数据保护的关键不是“核心表加密了没有”，而是敏感数据有没有在链路边角被顺手泄露。`

## 2.3.9 通讯安全：不要因为“是内网”就默认明文可信

这就是你提到当前稿子里明显遗漏的一项，而且它非常值得单独讲。

### PDF 核心检查项原文

> “为所有敏感信息采用加密传输。其中应该包括使用 TLS 对连接的保护，以及支持对敏感文件或非基于 HTTP 连接的不连续加密。”
>
> “TLS 证书应当是有效的，有正确且未过期的域名，并且在需要时，可以和中间证书一起安装。”
>
> “没有成功的 TLS 连接不应当后退成为一个不安全的连接。”
>
> “为所有要求身份验证的访问内容和所有其他的敏感信息提供 TLS 连接。”

OWASP 这章的重点并不复杂：

- 敏感信息传输要用 TLS。
- 证书要有效、域名要正确、过期要可监控。
- TLS 失败时不能静默回退到不安全连接。
- 只要涉及认证、敏感数据、外部系统对接，就应该默认使用受保护的连接。

这一章对今天的团队尤其有现实感，因为最容易被忽略的恰恰是这些“不是公网入口”的地方：

- 网关到后端之间默认明文。
- 后端到数据库、消息队列、缓存、中间件之间默认明文。
- 回调接口、Webhook、第三方对接只验业务参数，不认真校验证书和通道。
- 证书到期依赖人工记忆，没有自动监控和演练。

如果这一部分只留一句话：

`通讯安全不是“外网才需要”，而是所有敏感链路都应该默认加密、默认校验、默认不降级。`

## 2.3.10 系统配置：很多风险根本不是代码写出来的

这章特别适合拿来提醒大家，安全底线不仅在业务代码里，也在运行环境里。

### PDF 核心检查项原文

> “确保服务器、框架和系统部件采用了认可的最新版本。”
>
> “确保服务器、框架和系统部件安装了当前使用版本的所有补丁。”
>
> “移除所有不需要的功能和文件。”
>
> “移除在 HTTP 相应报头中有关 OS、Web 服务版本和应用程序框架的无关信息。”

最值得讲的点包括：

- 服务器、框架、组件、镜像要持续打补丁。
- 删除不需要的功能、文件、示例页、测试页、默认账号。
- 关闭目录列表、限制多余 HTTP 方法、隐藏版本信息。
- 对管理端点、调试能力、文档页面、诊断页面做环境隔离。

对我们团队最现实的落点是：

- Spring Boot Actuator 暴露过多端点。
- Swagger、H2 Console、测试页、调试页跑到了生产。
- Nginx、Tomcat、网关把详细版本信息直接暴露出去。
- 默认配置“先开着再说”，最后没人收口。

如果这一部分只留一句话：

`很多配置型漏洞不是因为没人会修，而是因为它们长期被当成“上线后再说的小事”。`

## 2.3.11 数据库安全：别让数据库成为“最后没人守的一层”

这章不能只讲“防 SQL 注入”，那样会讲窄。

### PDF 核心检查项原文

> “使用强类型的参数化查询方法。”
>
> “当应用程序访问数据库时，应使用尽可能最低的权限。”
>
> “连接字符串不应当在应用程序中硬编码。连接字符串应当存储在一个可信服务器的独立配置文件中，并且应当被加密。”
>
> “应用程序应当以不同的凭证为每个信任的角色（比如：用户、只读用户、访问用户、管理员）连接数据库。”

对团队最重要的点其实有四类：

- 查询要参数化，少让字符串拼接进入 SQL。
- 账号权限要最小化，不要让应用账号天然拥有过大权限。
- 连接串、凭据、密钥不要硬编码在代码和配置仓库里。
- 存储过程、动态 SQL、批量导入导出、备份恢复同样是安全面。

最适合拿来讲的现实例子有：

- MyBatis 使用 `${}` 直接拼条件。
- 应用使用 `root` 或高权限账号连库。
- 生产库账号同时拥有读、写、删、DDL 甚至管理权限。
- 备份文件落在共享目录里，权限长期无人管理。

如果这一部分只留一句话：

`数据库安全不只是“别被注入”，还包括别把高权限、明文凭据和危险操作变成默认状态。`

## 2.3.12 文件管理：文件上传、下载、导入、导出都要当高风险能力设计

很多团队会把文件处理当成功能细节，但 OWASP 把它单独列出来是有道理的。

### PDF 核心检查项原文

> “在允许上传一个文档以前进行身份验证。”
>
> “只允许上传满足业务需要的相关文档类型。”
>
> “通过检查文件报头信息，验证上传文档是否是所期待的类型。只验证文件类型扩展是不够的。”
>
> “关闭在文件上传目录的运行权限。”

这章最值得重点提醒的是：

- 不能只看扩展名，要同时看 MIME、文件头、大小、内容类型。
- 上传后的文件不要保留原始文件名，不要落在可执行路径下。
- 下载、预览、导出接口同样要做权限校验和路径约束。
- 对象存储、共享目录、临时目录都要考虑默认权限和生命周期。

最常见的事故来源其实很朴素：

- 页面限制上传 `.jpg`，后端只看扩展名。
- 上传目录可执行，或者和静态资源目录混用。
- 导出文件名、对象 key、下载路径可控，导致覆盖、探测或穿越。
- 文件本体权限没问题，但访问 URL 默认公开。

如果这一部分只留一句话：

`文件能力不是“附属功能”，而是集输入、存储、权限、执行风险于一体的高危入口。`

## 2.3.13 通用编码规范：安全不是只靠几个“安全功能点”

这一章适合拿来收束前面所有内容，因为它讲的是“编码时默认该怎么做”。

### PDF 核心检查项原文

> “为常用的任务使用已测试且已认可的托管代码，而不创建新的非托管代码。”
>
> “使用特定任务的内置 API 以执行操作系统的任务。不允许应用程序直接将代码发送给操作系统，特别是通过使用应用程序初始的命令 shell。”
>
> “不要将用户提供的数据传递给任何动态运行的功能。”
>
> “审核所有从属的应用程序、第三方代码和库文件，以确定业务的需要，并验证功能的安全性，因为它们可能产生新的漏洞。”

最适合提炼给团队的点包括：

- 优先使用成熟框架、库和平台能力，不要重复造安全轮子。
- 安全控制尽量集中化、可复用、可测试，而不是散落在各个接口里。
- 不要硬编码凭据、后门开关、调试逻辑、临时绕过逻辑。
- 失败时要倾向于安全失败，而不是异常时默认放行。

这章适合配合一句比较工程化的话来讲：

`真正可持续的安全，不靠开发者每次都临场发挥，而靠默认模板、统一组件、自动检查和集中收口。`

## 2.4 第一部分的小结

如果把第一部分压缩成十条团队底线，我建议是：

1. 所有输入都不可信，服务端统一校验。
2. 所有输出都按上下文编码。
3. 认证、会话、登出、找回密码要按完整链路设计。
4. 授权必须做到接口层和数据层，而不是只做页面层。
5. 任何 SQL、命令、模板、表达式拼接都要高度警惕。
6. 敏感数据、密钥、证书、连接串不能明文散落。
7. 错误对外收敛，日志对内完整。
8. 文件上传、下载、导入、导出都要当高风险能力设计。
9. 默认配置不可信，敏感链路默认加密，安全基线要自动化。
10. 安全不是上线前扫一次，而是需求到部署全过程。

# 三、第二部分：`OWASP Top 10:2025` 与我们团队最相关的风险

## 3.1 一个先说明的事实

> [OWASP Top 10:2025](https://owasp.org/Top10/2025/A01_2025-Broken_Access_Control/)

截至 `2026-05-26`，OWASP 官方已经发布 `OWASP Top 10:2025`。这意味着我们今天不能再只用 2021 的框架来理解当前的主流风险视角。

2025 版 Top 10 排名如下：

1. `A01 Broken Access Control`
2. `A02 Security Misconfiguration`
3. `A03 Software Supply Chain Failures`
4. `A04 Cryptographic Failures`
5. `A05 Injection`
6. `A06 Insecure Design`
7. `A07 Authentication Failures`
8. `A08 Software or Data Integrity Failures`
9. `A09 Security Logging & Alerting Failures`
10. `A10 Mishandling of Exceptional Conditions`

## 3.2 2025 版最值得强调的三个变化

### 变化一：安全配置错误升得很高

`A02 Security Misconfiguration` 排到第 2 位，说明今天大量真实问题来自配置、暴露面和缺少加固，而不是只有编码错误。

### 变化二：软件供应链成为核心风险

`A03 Software Supply Chain Failures` 排到第 3 位，说明现代应用的安全边界已经远不止“我们自己写的代码”。

### 变化三：异常条件处理不当进入 Top 10

`A10 Mishandling of Exceptional Conditions` 的出现很有代表性。它提醒我们：失败路径本身就是攻击面，很多事故发生在超时、空值、回退、异常兜底这些地方。

## 3.3 对我们团队最值得详细讲的六项

如果要在 1 小时里兼顾完整度和重点，我建议详细讲六项，剩余四项做短讲。

## 3.3.1 A01 Broken Access Control

这是最应该排第一位讲给团队听的风险。

它的核心不是“系统有没有登录”，而是：

- 用户能不能访问不属于自己的资源
- 用户能不能执行不该执行的操作
- 系统有没有把对象级、字段级、租户级边界守住

在我们团队里，最常见的表现是：

- 改 ID 看别人数据
- 菜单隐藏了，但接口仍能调
- 审批、导出、下载、报表接口漏了数据权限
- 多租户隔离只做在页面上，没有做在服务端查询上

防范这类风险，最关键的不是“多加几个 if”，而是建立统一授权模型：

- 统一身份
- 统一角色/权限模型
- 统一资源归属判断
- 关键对象的 ownership 校验
- 越权专项测试

一句话总结：

`越权问题本质上是“业务边界没有被系统真正执行”。`

## 3.3.2 A02 Security Misconfiguration

这类问题特别适合讲给运维和后端，也必须让前端知道它为什么重要。

它常见于：

- 默认配置没关
- 不必要的服务、页面、端点对外暴露
- 错误页暴露细节
- 安全头缺失
- 云资源、对象存储、容器、网关没有做收口

对我们团队最现实的例子：

- Spring Boot Actuator 暴露太多端点
- Swagger 或测试接口跑到了生产
- Nginx/Tomcat 暴露版本信息
- 对象存储桶或文件服务默认公开

这类问题最麻烦的地方是，它常常不是某个开发同学“写错了一行代码”，而是整个交付链路缺少统一安全基线。

所以它的治理方式必须偏工程化：

- 基线模板
- 自动检查
- 环境差异收敛
- 暴露面盘点
- 上线前配置核查

## 3.3.3 A03 Software Supply Chain Failures

这项排名升到第 3，非常值得团队重视。

过去大家常说“不要引入有漏洞的依赖”，但今天的供应链风险已经远不止依赖版本：

- Maven 传递依赖
- 镜像来源
- 制品仓库
- CI/CD 权限
- 发布流程
- 插件、脚本、自动化工具

对我们团队来说，最现实的几个问题是：

- 依赖长期无人盘点
- 开发机、CI、制品库权限过大
- 生产包不是同一份构建产物
- 发布链路没有完整留痕

这一项和传统“安全编码”最大的区别在于：

`哪怕业务代码写得很好，构建和交付链路被污染，结果仍然可能是不安全的。`

所以比较关键的治理动作包括：

- 做依赖和镜像清单
- 关注高危漏洞公告
- 构建产物可追溯
- 对关键仓库和流水线做最小权限与 MFA
- 逐步引入 SBOM、签名和来源证明

## 3.3.4 A05 Injection

虽然注入在 2025 版不再排第 1，但它仍然是最容易被理解、也最容易真的出事的一类风险。

对我们团队来说，注入绝不只等于 SQL 注入，还包括：

- SQL / HQL / JPQL 注入
- 命令注入
- 表达式注入
- 模板注入
- 未来 LLM 场景下的 Prompt Injection

最常见的错误心智是：

`我们平时都用 ORM，所以应该不会有注入了。`

但实际风险仍然很多：

- MyBatis `${}` 动态拼接
- 排序字段、表名、过滤表达式直接传入
- shell 命令拼接
- 表达式引擎或脚本执行能力暴露

防范注入问题有一个很稳定的思路：

- 能不用解释器式接口就不用
- 能结构化传参就不要字符串拼接
- 必要时用白名单约束可变部分
- 对动态能力做专门审计

一句话总结：

`注入的本质，是把“不可信输入”当成“程序指令的一部分”。`

## 3.3.5 A06 Insecure Design

这项风险很适合拿来提醒团队：安全不只是实现问题，更是设计问题。

它关注的是：

- 安全控制有没有被设计出来
- 业务流程有没有滥用路径
- 系统是不是只为“正常用户”设计，而没有为“恶意行为”设计

现实中，这类问题经常体现在：

- 密码找回逻辑过弱
- 审批流或退款流可以被绕过
- 多租户隔离先天就薄弱
- 抢购、领券、接口批量调用没有反滥用设计

这一项特别值得带给团队一个观念：

`很多严重问题不是因为代码写错，而是因为设计阶段根本没把攻击者放进场景里。`

所以在需求评审和方案评审阶段，就应该开始问：

- 如果有人恶意使用这个流程，会怎样？
- 如果有人绕开前端直接打接口，会怎样？
- 如果鉴权组件失败了，会怎样？

## 3.3.6 A10 Mishandling of Exceptional Conditions

这是 2025 版新增的类别，也是我特别建议重点讲的一项。

很多系统平时测试的是“正常流程”，但真实事故更容易出在：

- 参数缺失
- 空值
- 超时
- 重试
- 权限组件失败
- 网络异常
- 并发竞争
- 回滚不完整

为什么这项值得强调？

因为今天我们的系统越来越依赖：

- 微服务
- 网关
- 缓存
- MQ
- 外部 API
- AI 服务

一旦某个环节失败，如果系统用的是“先放过、先降级、先返回默认值”这种思路，就可能直接把安全控制绕开。

这一项我最建议大家记住一句话：

`关键安全能力在异常情况下必须 fail-secure，而不是 fail-open。`

## 3.4 其余四项的短讲版本

## A04 Cryptographic Failures

重点不是“有没有加密”，而是：

- 算法是否过时
- 密钥是否硬编码
- 是否用错场景
- 是否真正保护了敏感数据和传输链路

## A07 Authentication Failures

重点不是“能不能登录”，而是：

- 是否抗弱密码、撞库、暴力破解
- 找回密码是否过弱
- 会话/token 生命周期是否合理
- 管理系统是否启用 MFA

## A08 Software or Data Integrity Failures

重点是：

- 我们是否信任了不该信任的代码、制品、数据或更新渠道
- 是否验证来源和完整性

## A09 Security Logging & Alerting Failures

重点是：

- 有没有把关键安全事件记录下来
- 能不能及时发现异常
- 日志是不是会泄露更多问题

## 3.5 第二部分的小结

如果把 Top 10:2025 讲给我们团队，我最希望大家记住的不是排名，而是三层变化：

- 第一层，经典风险仍然存在，尤其是越权、注入、认证。
- 第二层，工程化风险越来越重要，尤其是配置、供应链、完整性、日志。
- 第三层，失败路径已经正式成为攻击面，异常处理不能再只当稳定性问题看。

# 四、第三部分：AI 时代，OWASP 发生了什么变化

## 4.1 一个总体判断

OWASP 正在从“经典 Web 安全知识库”扩展成“现代软件与 AI 系统安全体系”。

这不是多加了几个 AI 术语，而是 OWASP 的关注范围真的变了：

- 从 Web 应用扩展到 LLM、Agent、MCP、Skill
- 从编码缺陷扩展到协议、身份、供应链、治理
- 从指南扩展到实战资源、清单、标准化和生态项目

## 4.2 最值得团队知道的几个变化

## 4.2.1 OWASP Top 10 for LLM 已升级为 GenAI Security Project

> https://genai.owasp.org/

这说明 OWASP 不再把 AI 安全理解成“给 LLM 列 10 条风险”这么简单，而是开始建设一整套持续治理体系。

这对团队的启发是：

`只要我们开始接入大模型，无论是做问答、RAG、代码助手、文档处理还是 Agent，我们就已经进入一个新的安全面。`

## 4.2.2 LLM Top 10:2025

> https://genai.owasp.org/llm-top-10/

如果只把 AI 风险理解成“幻觉”，那会把问题看窄很多。

按 OWASP 官方 `LLM Top 10:2025` 的口径，所谓“幻觉”更接近 `LLM09 Misinformation` 这一个风险点里的部分表现，但真正的安全面已经同时覆盖：

- 输入会不会操纵模型
- 敏感数据会不会被模型吐出来
- 模型、数据、LoRA、依赖和平台会不会被供应链污染
- 模型输出会不会直接触发下游危险动作
- Agent 会不会因为权限过大而替攻击者做事
- 向量库、Embedding、RAG 会不会变成新的泄露面
- token、算力、费用会不会被无限消耗

换句话说，`AI 风险不是只有“答错”，而是“会不会被操纵、会不会泄露、会不会乱执行、会不会失控”。`

### 2025 Top 10 全量概括

### LLM01:2025 Prompt Injection

- 是什么：攻击者通过用户输入、网页、文件、图片等内容改变模型行为，让模型偏离原本指令。
- 为什么：它可能导致越权调用工具、泄露系统提示词、泄露数据、操纵决策，甚至驱动下游系统执行危险动作。
- 怎么防：约束模型职责边界，隔离外部内容，严格校验输入输出，对高风险动作做人审或二次确认，不把安全控制只寄托在 prompt 上。

### LLM02:2025 Sensitive Information Disclosure

- 是什么：模型在输出中泄露个人信息、业务敏感信息、凭据、训练数据片段、内部配置或专有知识。
- 为什么：LLM 会把训练数据、上下文数据、RAG 检索结果和系统信息混在一起生成回答，一旦边界没控好，就会把“不该说的”说出来。
- 怎么防：做脱敏和红线识别，限制模型可见数据范围，避免默认把用户输入用于训练，清晰声明数据使用方式，并对输出做敏感信息检测。

### LLM03:2025 Supply Chain

- 是什么：风险不只来自业务代码，也来自模型、数据集、向量库、LoRA 适配器、开源依赖、镜像、训练平台和推理平台。
- 为什么：第三方模型和数据很难像普通代码一样完全审计，一旦被投毒、篡改或替换，模型行为就可能整体失真，甚至带入后门。
- 怎么防：只用可信来源，维护 SBOM/AIBOM 或模型清单，做签名和哈希校验，定期补丁与重评估，对供应商、模型和数据做安全评估与红队验证。

### LLM04:2025 Data and Model Poisoning

- 是什么：攻击者污染预训练、微调、Embedding 或知识库数据，让模型学到偏置、后门或错误事实。
- 为什么：这会直接影响模型完整性，让模型在特定触发条件下输出被操纵的结果。
- 怎么防：跟踪数据来源和版本，严格筛选训练与知识库数据，对异常样本做检测，做鲁棒性测试和红队验证。

### LLM05:2025 Improper Output Handling

- 是什么：系统把 LLM 输出当成可信内容，未经校验就传给页面、脚本、SQL、命令、邮件模板或其他下游系统。
- 为什么：LLM 输出本质上仍然是不可信输入，只是来源从“用户”变成了“模型”，一样可能引发 XSS、注入、命令执行或流程劫持。
- 怎么防：把模型当成不可信协作者，对输出做校验、编码和白名单约束，高风险下游只接受结构化和可验证格式。

### LLM06:2025 Excessive Agency

- 是什么：给 LLM 或 Agent 过大的工具权限、数据权限和执行权限，让它能“直接做事”而不是“只负责建议”。
- 为什么：一旦模型被误导、越狱或理解错误，过大的 agency 会把问题从“说错”升级成“做错”。
- 怎么防：最小权限、任务级授权、敏感操作审批、工具分级、审计日志和可撤销机制。

### LLM07:2025 System Prompt Leakage

- 是什么：系统提示词、本地规则、隐藏指令或其中包含的敏感信息被模型泄露给用户。
- 为什么：很多团队误以为 system prompt 天然保密，但它常常能被套问、抽取、拼接或侧信道方式暴露。
- 怎么防：不要把密钥、凭据、权限规则藏在 prompt 里；把关键控制放到确定性代码里；减少 prompt 中的敏感内容并做隔离。

### LLM08:2025 Vector and Embedding Weaknesses

- 是什么：向量库、Embedding、RAG 检索链路在权限隔离、数据完整性、检索准确性上存在缺陷。
- 为什么：它会导致跨租户检索、知识库投毒、错误召回和隐私泄露，尤其在多租户和企业知识库场景里非常现实。
- 怎么防：对向量数据做权限隔离和来源治理，控制写入链路，监控检索质量，对知识库变更做审核和追踪。

### LLM09:2025 Misinformation

- 是什么：模型生成看似可信但错误、偏置、误导甚至危险的信息。
- 为什么：它会影响用户判断、业务决策、客服答复、医疗金融建议和代码生成质量，所以“幻觉”本身也会演变成安全与合规问题。
- 怎么防：高风险场景引入事实校验、来源引用、人工复核、领域限制和明确免责声明，不让模型单独承担关键决策。

### LLM10:2025 Unbounded Consumption

- 是什么：攻击者通过超长输入、高频请求、复杂查询或模型提取行为，持续消耗 token、算力、费用和服务能力。
- 为什么：LLM 的成本和资源消耗远高于普通接口，所以这类问题会同时带来 DoS、拒绝钱包和模型能力被“薅走”的风险。
- 怎么防：限流、配额、超时、输入长度限制、资源监控、异常检测、优雅降级和高成本操作隔离。

### 其中最值得详细讲的 Top 3

#### 4.2.2.1 LLM01 Prompt Injection：AI 时代的“输入污染”

OWASP 官方把它放在第 1 位，非常合理。因为它几乎是所有 LLM 应用的基础风险。

它最核心的意思是：

`用户输入、外部内容、附件、网页、图片、知识库文本，都可能不只是“数据”，而是在偷偷给模型下指令。`

这件事和传统注入有点像，但又不完全一样。传统注入通常是“把输入拼进解释器”；Prompt Injection 则是“让模型把攻击者的话当成更高优先级的指令去理解”。

它至少有两种常见形态：

- Direct Prompt Injection：攻击者直接在对话里要求模型忽略原规则、泄露提示词、调用工具。
- Indirect Prompt Injection：攻击者把恶意指令藏进网页、文档、邮件、图片 OCR 文本、RAG 知识片段，让模型在读取这些内容时被带偏。

为什么它危险？因为一旦模型后面连着：

- 检索系统
- 工作流引擎
- 发邮件能力
- 下单/审批能力
- 文件系统
- Shell 或其他工具

那么“被带偏”就不再只是答错一句话，而可能升级成：

- 泄露内部信息
- 误发消息
- 错误执行操作
- 越权访问外部系统
- 影响审批或业务判断

官方 mitigation 里有几个非常值得带给团队：

- 不要把 system prompt 当成唯一防线，要用确定性代码收口关键控制。
- 明确定义模型的角色、能力边界和禁止事项。
- 让高风险输出必须满足固定格式，并由程序校验后才能进入下游。
- 对输入和输出同时做过滤，不只是防用户输入，也防模型输出“夹带指令”。
- 对高风险动作引入人工确认，不要让模型一句话就直接执行。

如果把它翻译成团队能马上理解的话，就是：

`只要模型会读外部内容、会用工具、会驱动流程，Prompt Injection 就应该被当成默认存在的风险。`

#### 4.2.2.2 LLM02 Sensitive Information Disclosure：AI 会把“不该说的”说出来

这项特别值得讲，因为很多团队接入 LLM 时，第一反应是“先把更多数据喂进去，让它更聪明”。但安全上最怕的恰恰是这一步。

官方定义下，敏感信息泄露不只是个人隐私，还包括：

- PII
- 财务信息
- 健康信息
- 业务机密
- 安全凭据
- 法律文档
- 模型训练方法
- 源码和系统配置

为什么 LLM 场景容易出现这个问题？

- 用户可能把敏感内容直接输入给模型。
- RAG 可能把原本只该内部可见的文档检索出来。
- 系统提示词、错误信息、上下文拼接可能暴露内部细节。
- 平台可能默认保留数据，甚至把数据用于后续训练或改进。

对团队来说，最现实的事故形态通常不是“黑客突破了数据库”，而是：

- 员工把合同、代码、账号信息贴进外部模型
- AI 助手把其他租户或其他用户的数据答出来
- 模型在长上下文里回显内部提示词或敏感字段
- 调试日志、错误页、工具返回结果把凭据一起带出来

防范这类问题，官方给出的方向很明确：

- 先做数据脱敏和输入校验，再决定什么数据允许进模型。
- 用最小权限限制模型在运行时能看到哪些数据源。
- 明确告知用户数据会不会被保留、用于训练、多久删除，以及是否支持退出。
- 对输出做敏感信息检测、tokenization、redaction，不要只相信 prompt 里的“请不要泄露”。
- 不要把内部配置、系统前导规则、凭据放进容易被拼接或回显的上下文。

这一项最适合留给团队的一句话是：

`接入 LLM 之后，数据泄露的路径不再只有“存储”和“传输”，还多了一个“生成”。`

#### 4.2.2.3 LLM03 Supply Chain：模型世界的供应链，比传统软件更宽

这一项排到第 3，非常说明问题。它提醒我们：AI 系统的可信边界远不只是“我们自己写的业务代码”。

在 LLM 场景里，供应链至少包括：

- 基础模型
- 微调模型
- 训练数据集
- RAG 知识数据
- 向量库组件
- LoRA / PEFT 适配器
- 推理框架
- Python/Node 依赖
- 镜像、容器、CI/CD 和模型仓库

为什么这项风险更麻烦？

因为传统依赖至少还能看源码、做 SCA、打补丁；但模型和数据很多时候像“黑盒制品”，团队对其来源、篡改情况、训练方式、许可证约束、隐藏后门并没有同等可见性。

官方文档里点得很具体，值得我们讲给团队：

- 过期或无人维护的模型本身就是风险。
- 预训练模型可能带有偏置、后门或被直接篡改。
- 模型来源证明很弱，Model Card 只能提供说明，不能真正证明未被掉包。
- LoRA 适配器、模型合并、格式转换服务、共享训练环境，都会引入新的污染点。
- 供应商 T&Cs 和隐私政策变化，也可能让你的业务数据被拿去训练。

比较稳妥的治理方式应该是工程化的，而不是“靠大家小心一点”：

- 只从可验证来源拉取模型、适配器和数据，并校验签名、哈希或完整性。
- 维护依赖清单、模型清单、数据清单，逐步走向 SBOM / AIBOM / ML-BOM。
- 对第三方模型和适配器做专项评估、红队测试和持续重评估。
- 把补丁、下线、替换、许可证审查纳入正式流程，而不是一次性接入。
- 对模型仓库、转换服务、训练环境、推理环境做最小权限和审计监控。

如果只留一句话给团队，我建议是：

`AI 供应链风险的本质，是你信任了很多自己并没有真正验证过的模型、数据和平台。`

## 4.2.3 Agentic Skills Top 10

> https://owasp.org/www-project-agentic-skills-top-10/

OWASP 官方对 skill 的定位非常清楚：它不是单个工具，不只是 prompt 片段，而是一种可复用的“行为模板”或“工作流模板”。它往往会同时定义：

- 任务拆解方式
- 工具调用顺序
- 文件、网络、Shell 的使用方式
- 安全护栏和输出格式
- 持久记忆和跨会话状态

这也是它为什么值得单独列一个 Top 10。因为 skill 出问题时，被影响的已经不是“回答质量”，而是：

- agent 会不会被教成恶意行为
- 工具权限会不会被放大
- 文件和凭据会不会被读走
- 自动化流程会不会被篡改
- 风险会不会跨平台复用和扩散

可以把它翻译成一句更适合讲给团队的话：

`MCP 关注“能做什么”，Skill 关注“具体怎么做”；Skill 一旦被污染，污染的是整个执行流程。`

### Agentic Skills Top 10 全量概括

### AST01 Malicious Skills

- 是什么：攻击者发布看起来正常、实则内含恶意载荷的 skill，可能在说明文档、配置、脚本或依赖里藏后门。
- 为什么：skill 往往继承 agent 的真实权限，一旦安装成功，就可能直接接触 SSH key、API key、浏览器数据、钱包文件、Shell 和网络能力。
- 怎么防：只允许可信发布源，要求签名和注册表扫描，在发布和安装时做行为分析，把 skill 运行放进沙箱，并建立信誉与审计机制。

### AST02 Supply Chain Compromise

- 是什么：skill 注册表、发布流程、依赖树、配置文件和维护者账号被污染，导致看似正常的 skill 通过供应链进入环境。
- 为什么：skill 生态还没有像 npm、PyPI 那样成熟的来源证明和透明日志，一旦上游被劫持，下游 agent 会直接继承被污染结果。
- 怎么防：做来源证明、透明日志、递归依赖扫描、哈希锁定和内部 allowlist，把 skill 配置文件也当成可执行面来审查。

### AST03 Over-Privileged Skills

- 是什么：skill 拿到的文件、网络、数据库、Shell、记忆文件写入等权限，明显大于它完成任务实际所需。
- 为什么：一旦 skill 被 prompt injection、误操作或恶意输入驱动，过大权限会把本来局部的问题放大成整体破坏。
- 怎么防：要求声明权限清单，使用最小权限和按 skill 隔离的凭据，对运行时真实行为做校验，并对高危权限做额外审核。

### AST04 Insecure Metadata

- 是什么：skill 的名称、作者、权限声明、风险等级、依赖说明等元数据可被伪造、弱化或伪装。
- 为什么：用户和平台常常首先相信元数据，一旦它被操纵，就会出现品牌冒充、权限低报、风险误导和搜索投毒。
- 怎么防：对元数据做静态分析和格式校验，对品牌和发布者做校验，把声明权限与真实行为比对，并展示元数据来源与签名信息。

### AST05 Unsafe Deserialization

- 是什么：skill 加载过程会解析 YAML、JSON、Markdown、依赖文件等，如果解析器或加载流程不安全，可能在加载时就执行恶意内容。
- 为什么：skill 的攻击面不只在运行阶段，也在“被安装、被解析、被初始化”的那一刻，而且往往是自动发生的。
- 怎么防：使用安全解析器，限制允许字段，在隔离环境中解析和安装依赖，把 skill 文件当成不可信输入而不是静态配置。

### AST06 Weak Isolation

- 是什么：skill 与宿主 agent 运行在同一安全上下文，直接共享主机文件系统、网络、Shell 和长期凭据。
- 为什么：没有隔离时，任何一个 skill 的问题都可能直接变成整机问题，沙箱、容器、命名空间这些边界等于不存在。
- 怎么防：默认容器化或沙箱化执行，限制宿主模式，隔离每个 skill 的进程和资源，并对本地控制接口做认证、限流和最小暴露。

### AST07 Update Drift

- 是什么：已安装的 skill 长期不更新，或自动更新机制盲目信任上游，导致要么补丁滞后，要么恶意更新被自动吃进来。
- 为什么：skill 常由个人自行安装，缺少企业补丁管理；而且“新版本”本身如果没有签名和哈希，也未必值得信任。
- 怎么防：使用不可变哈希固定版本，对每次更新都做签名校验，生产环境禁用热更新，订阅安全公告并维护安装清单。

### AST08 Poor Scanning

- 是什么：传统代码扫描器很难识别 skill 里的自然语言恶意意图、上下文依赖行为和隐藏在文档中的指令。
- 为什么：skill 既有代码层，又有说明层；攻击者完全可以不写危险代码，只用文字诱导 agent 去做危险动作。
- 怎么防：同时扫描代码层和自然语言层，用语义分析加行为沙箱的多工具流水线，不把单一扫描结果当成最终信任依据。

### AST09 No Governance

- 是什么：组织里没有 skill 资产清单、审批流程、审计日志、撤销机制和责任归属，skill 安装与使用处于“影子 AI”状态。
- 为什么：安全团队看不见 skill，运维流程管不到 skill，离职、事故、审计、合规都很难收口。
- 怎么防：建立集中 inventory、审批和审计，把 skill 接入 CMDB/ITSM/CASB，对 agent 使用非人身份和生命周期管理。

### AST10 Cross-Platform Reuse

- 是什么：同一个 skill 在 OpenClaw、Claude Code、Cursor、VS Code 等平台间迁移时，安全属性没有被等价保留。
- 为什么：不同平台的 skill 格式、权限模型和风险字段不统一，迁移时很容易把关键安全信息“翻译丢了”。
- 怎么防：迁移时重新做安全校验，推动统一 skill 格式或字段映射，建立跨注册表威胁情报共享和平台无关的扫描能力。

### 其中最值得详细讲的 Top 3

#### 4.2.3.1 AST01 Malicious Skills：恶意 skill 不是“插件有 bug”，而是“插件本身就是恶意的”

OWASP 把这一项放在第 1 位，而且给的是 `Critical`，这是非常值得团队重视的信号。

它最核心的意思是：

`攻击者完全可以把恶意能力包装成一个“很好用的 skill”，让开发者自己安装进来。`

这和传统恶意依赖很像，但又更危险。因为 skill 不只包含代码，还可能包含：

- Markdown 里的自然语言指令
- 安装前置步骤
- Shell 脚本
- 依赖文件
- 记忆文件写入逻辑
- 与 agent 身份文件的交互

也就是说，攻击者不一定非要靠一段明显恶意的二进制代码，他还可以靠“写给 agent 看的说明文字”来诱导 agent 做坏事。

官方页面特别强调了 skill 的“双重攻击面”：

- 代码层攻击面：脚本、依赖、下载器、反连、后门。
- 自然语言攻击面：在 `SKILL.md` 或说明文字里藏社工指令、越权操作、环境准备命令。

这类风险为什么对团队很现实？

- skill 往往带着主机权限运行。
- skill 可能直接读本地代码、配置、SSH key、浏览器数据。
- skill 常常具备网络出口，能把数据带走。
- skill 还可能写 agent 的记忆文件，形成持久化后门。

换成团队能立刻代入的场景，就是：

- 一个“Google Workspace 助手”其实在偷 `.env`
- 一个“代码整理工具”在安装前要求执行攻击者给的脚本
- 一个“总结文档 skill”把恶意指令写进 agent 的 `MEMORY.md`

官方 mitigation 有几个很值得直接讲：

- skill 发布和安装都应要求加密签名校验，而不是只看名称和 star。
- 注册表要做行为扫描，不是只做关键字匹配。
- skill 执行要放进容器或沙箱，不要默认跑在宿主机。
- 任何“Prerequisites”或安装前命令都不能自动执行，必须显式审查。
- skill 需要有信誉、扫描状态和发布者信任等级，而不是所有发布者一视同仁。

如果只留一句话给团队，我建议是：

`在 agent 生态里，恶意 skill 更像“会伪装成生产力工具的恶意员工”，不是普通的配置错误。`

#### 4.2.3.2 AST02 Supply Chain Compromise：skill 供应链已经不只是“依赖版本问题”

这一项同样是 `Critical`，而且它和传统软件供应链相比，复杂度更高。

在 skill 场景里，供应链至少包括：

- skill 注册表
- skill 发布账号
- `SKILL.md` 与 manifest
- `package.json`、`requirements.txt` 等依赖树
- repo 里的 hook、settings、环境变量配置
- skill 更新通道

为什么它更难防？

因为很多 skill 生态的发布门槛很低，但权限继承却很高。一个看起来无害的 skill，一旦被 agent 安装，拿到的可能不是“包管理器权限”，而是：

- 本地文件访问
- API key
- shell 执行能力
- 外部网络通信能力
- 与其他工具链的联动能力

官方页面特别提到一个很值得带给团队的点：

`配置文件已经不再只是静态配置，而可能变成执行路径本身。`

这意味着我们不能只盯业务代码和依赖包，还要把这些也当成供应链检查对象：

- `.claude/settings.json`
- hooks
- agent 配置文件
- 环境覆盖项
- skill 初始化时自动触发的依赖安装

对团队来说，这类风险的典型形态包括：

- 顶层 skill 看起来很干净，真正恶意 payload 藏在嵌套依赖里
- 发布者账号被盗，推送一个“修复版”但其实是后门版
- 恶意 repo 被 clone 后，仅仅因为“打开项目”就触发配置执行

比较稳妥的治理思路应该是：

- skill 和发布者要有明确来源证明，能追到签名身份。
- 发布、更新、删除等注册表动作要有透明日志。
- 递归依赖要做扫描，不能只扫最外层 skill。
- 版本固定要落到不可变哈希，而不是只写版本号范围。
- 企业内部优先使用 allowlist 或内部镜像，不直接信任公网注册表。

这一项最适合留一句：

`在 agent 时代，供应链被污染时，进入环境的不是一个“有漏洞的包”，而是一整套带执行能力的行为模板。`

#### 4.2.3.3 AST03 Over-Privileged Skills：权限过大，会把普通失误放大成严重事故

这一项排在第 3，而且非常符合团队日常直觉。

它不是说 skill 一定是恶意的，而是说：

`就算 skill 初衷是好的，只要它拿到的权限太大，后果也会很糟。`

OWASP 官方的一个关键点特别值得讲给团队：skill 的权限问题，不只是“工具能不能调”，还包括“自然语言意图和系统权限之间有没有错配”。

这和传统最小权限有什么不同？

传统系统里，开发者常常直接控制函数、接口和权限点；但在 skill 场景里，agent 会把自然语言目标翻译成一连串动作。如果底层只做了“允许调用这个工具”，却没约束“允许出于什么目的、对什么对象、在什么边界内调用”，那就会出现：

- 本来只是查天气的 skill，却能读整个 `.env`
- 本来只是看邮箱的 skill，却顺手执行删除邮件
- 本来只是查询数据库的 skill，却能执行写操作甚至破坏操作

这类风险为什么特别现实？

- 很多 agent 默认继承宿主的广泛权限。
- 很多 skill 没有独立权限清单或清单没人认真看。
- 一旦上游有 prompt injection，过大权限会立刻放大影响。
- 记忆文件、身份文件、配置目录这些敏感位置，很容易被顺手拿到写权限。

官方 mitigation 对我们团队很有借鉴价值：

- skill 必须声明权限清单，没有 manifest 的不应直接信任。
- 凭据要按 skill 作用域拆分，不能所有 skill 共用 agent 级大钥匙。
- 对 `SOUL.md`、`MEMORY.md`、`AGENTS.md` 这类身份和记忆文件的写权限要单独升高审查级别。
- 运行时要真正执行权限约束，而不是只在 manifest 里写一行。
- 网络权限要细到域名 allowlist，而不是简单的 `network: true`。
- 在沙箱测试里把“声明权限”和“真实行为”做一致性比对。

如果只留一句话，我建议是：

`过度授权的 skill，往往不是自己主动作恶，而是在别人一推、一骗、一诱导之后，替攻击者把事做成了。`

# 五、第四部分：如何让 AI 增强 OWASP 在项目中的参与度和使用便捷性

## 5.1 核心判断

AI 不应该替代安全专家，但非常适合做两件事：

- 把安全能力前移
- 把安全规范嵌入日常研发动作

也就是说，最值得做的不是“让 AI 帮我们扫所有漏洞”，而是：

`让 AI 在需求评审、方案评审、代码评审、测试点设计、上线检查这些高频动作里，顺手把 OWASP 带进来。`

## 5.2 你提出的 `OWASP agent skill`，是一个很好的切入点

我很认可这个方向，因为它有三个优势：

- 它贴近真实研发流程
- 它把静态规范变成可调用能力
- 它对团队成员的安全水平要求更友好

这个 skill 可以服务至少四种场景：

- 需求评审：看有没有越权、认证、数据保护、审计缺口
- 方案评审：看有没有不安全设计、信任边界错误、权限模型缺陷
- 代码评审：看有没有注入、日志泄露、文件上传、配置错误
- 上线检查：看有没有默认配置、暴露端点、弱凭据、缺少日志

## 5.3 一个更完整、也更容易落地的方案组合

我建议不要把它只理解成“做一个 skill”，而是理解成“一组让 OWASP 进入研发流程的 AI 能力”。

## 5.3.1 第一优先级：快速试点

### A. OWASP 方案评审助手

输入方案文档、接口文档、流程图，让 AI 从 OWASP 视角产出问题清单。

这一步最能前移 `A06 Insecure Design`。

### B. OWASP 代码审查助手

输入代码 diff 或模块代码，让 AI 对照 OWASP 给出风险点、原因和修复建议。

这一步最适合覆盖：

- 越权
- 注入
- 认证/会话
- 敏感信息日志
- 文件上传
- 异常 fail-open

### C. OWASP 安全问答 Copilot

让开发在日常工作中直接提问：

- “这个接口会不会有越权风险？”
- “JWT 这样设计是否合理？”
- “上传 Excel 应该防什么？”

这一类能力的价值在于持续培养安全意识，而不是只在上线前集中补课。

## 5.3.2 第二优先级：工程化增强

### D. OWASP 安全测试点生成助手

根据接口和业务流程，自动生成越权、注入、认证、文件上传、异常处理等测试点。

### E. OWASP 上线前基线检查助手

对配置、镜像、端点、证书、Header、日志和凭据做核查，重点覆盖 `A02/A03/A04/A08/A09`。

## 5.3.3 第三优先级：长期建设

### F. OWASP 知识库与模板库

把团队高频能力沉淀成安全模板：

- 登录接口模板
- 上传接口模板
- 导出接口模板
- 回调接口模板
- 管理台接口模板

这其实是在打造一条“安全的默认路径”。

## 5.4 `OWASP agent skill` 最小可行版本建议

如果分享之后要落一个试点，我建议先做一个最小版，不要一开始就追求覆盖全部 OWASP。

第一版可以先覆盖 8 到 12 条团队高频规则：

- 越权
- SQL 注入
- XSS / 输出编码
- 文件上传
- JWT 生命周期
- 敏感信息日志
- 密钥硬编码
- 默认配置暴露
- 异常 fail-open

skill 的输出建议固定成下面几类：

- 命中的 OWASP 类别
- 风险原因
- 可能的攻击方式
- 修复建议
- 推荐测试点
- 严重级别
- 是否需要人工复核

## 5.5 这一部分最想推动的三个行动项

### 行动项 1

选一个典型项目模块，试做 `OWASP code review skill`。

### 行动项 2

把 skill 的输出结果和团队现有 CR 习惯做对比，验证是否能提升 review 质量和一致性。

### 行动项 3

把经过验证的规则沉淀成团队自己的安全规范和模板库，让 AI 输出最终反哺制度资产。

## 5.6 第四部分的小结

我最希望这部分传达出去的一句话是：

`AI 最有价值的地方，不是替我们“负责安全”，而是让安全更早、更频繁、更低门槛地出现在每一次研发动作里。`

# 六、收尾：今天最值得记住的四句话

## 6.1 四句总结

1. `OWASP_zh.pdf` 虽然老，但它提供的是今天仍然有效的安全底线。
2. `OWASP Top 10:2025` 告诉我们，现代安全风险已经从“代码漏洞”扩展到“配置、供应链、异常处理和完整性”。
3. AI 时代的 OWASP 已经覆盖 LLM、Agent、MCP、Skill、非人身份和治理体系。
4. 最值得做的不是再多写一份规范，而是把 OWASP 通过 AI 和工程化能力嵌进日常研发流程。

## 6.2 一个适合结束时说的句子

`如果规范永远躺在文档里，它只能靠记忆执行；如果规范进入评审、进入代码、进入流水线、进入 agent，它才开始真正改变项目。`

# 七、附：讲述提示

## 7.1 讲这一场时的节奏建议

- 第一部分不要念清单，要讲“为什么这些规则今天还有效”。
- 第二部分不要平均用力，要重点讲 `A01/A02/A03/A05/A06/A10`。
- 第三部分不要堆新名词，要始终回答“这为什么和我们团队有关”。
- 第四部分不要讲成科幻，要给出能在团队里试点的最小动作。

## 7.2 哪些地方适合讲真实例子

- 越权：改一个 ID 就能看到别人的数据
- 注入：排序字段拼进 SQL
- 配置错误：测试端点出现在生产
- 文件上传：只看扩展名，不看文件头
- 异常处理：下游失败后默认放行
- AI 场景：给 agent 太大的 token 和工具权限

## 7.3 哪些地方可以和听众互动

- “你们见过最常见的越权问题是什么？”
- “大家所在项目里，生产环境有没有还开着 Swagger 或测试端点？”
- “如果以后项目接 LLM，最担心的是 prompt、token、工具，还是数据泄露？”
- “如果让团队下周就试点一个 AI 安全能力，最值得先做哪一个？”
