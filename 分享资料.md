# 第一部分：OWASP 安全编码规范

## 1.1 软件安全与风险原则

1. 为了保护业务免受来自与软件相关的不能接受的风险，了解风险的意义是很有帮助的。风险是一组威胁业务成功因素的集合。它可以被定义为：一个**威胁代理**与一个可能含有**漏洞**的**系统**交互，该漏洞可被**利用**并造成**影响**
2. 开发团队采用的方法和攻击者攻击应用程序所采用的方法之间有一个根本区别：
   * 开发团队通常采用的方法是基于应用程序的目的行为。换句话说，开发团队根据功能需求文档和用例设计一个应用程序以执行特定的任务
   * 而另一方面，攻击者，基于“没有具体说明应拒绝的行为，则被认为是可行的”原则，对于应用程序可以做什么更感兴趣

学习《OWASP 安全编码规范快速参考指南》的目的是培养我们的安全意识，让我们在方案设计、代码开发过程中能够思考系统“不能用来干什么”，而不仅仅是系统“需要用来干什么”

## 1.2 三个总原则

### 原则一：前端、客户端不可信

前端校验、隐藏字段、菜单隐藏、按钮置灰，这些限制都只能改善体验，不能替代安全控制，因为攻击者根本不会按页面走流程，他会直接改请求、改参数、改 Header、改 Cookie。



### 原则二：安全控制要集中化、服务端化、可复用

OWASP 在很多章节里都强调“集中控制”，这对团队尤其关键。集中化意味着安全代码可维护；服务端化意味着安全代码在安全环境中运行；可复用意味着安全代码在项目中是统一的、规范的



### 原则三：白名单 VS 黑名单

默认拒绝，而不是默认放行，即采用白名单校验



## 1.3 安全编码规范检查列表

### 1.3.1 输入验证

1. 识别所有的数据源，并将其分为可信的和不可信的。验证所有来自不可信数据源（比如：数据库、文件流等等）的数据
2. 在可信系统（比如：服务器）上执行所有的数据验证
3. 在处理以前，验证所有来自客户端的数据，包括：所有参数、URL、HTTP 头信息（比如：cookie 名字和数据值）
4. 验证正确的数据类型、数据范围、数据长度
5. 尽可能采用“白名单”形式，验证所有的输入
6. 应当为应用程序应提供一个集中的输入验证规则

注：

1. 现在 AI 会随手写一份已经存在的工具方法，而不是复用已有的工具方法，这是要注意的



### 1.3.2 输出编码

1. 在可信系统（比如：服务器）上执行所有的编码
2. 对于所有发送给客户端、且并非由应用程序内部产生的不可信数据，必须根据其最终出现的“上下文环境”进行针对性的编码处理。HTML 实体编码是一个例子，但不是在所有的情况下都适用（注：不要试图使用一种编码方式解决所有问题）

注：

1. 什么是上下文（Contextually）：

   * HTML 正文上下文：数据放在 `<div>数据</div>` 中。此时需要进行 HTML 实体编码（如将 `<` 转义为 `&lt;`），防止脚本注入
   * HTML 属性上下文：数据放在 `<input value="数据">` 中。除了 HTML 实体编码，还必须确保属性值被引号包裹，防止攻击者通过空格或闭合引号来注入新的属性（如 `onmouseover`）
   * JavaScript 上下文：数据放在 `<script> var name = "数据"; </script>` 中。此时 HTML 实体编码是无效的，必须使用 Unicode 转义编码（如 `\u003C`），否则攻击者可以轻易闭合字符串并执行任意代码
   * URL 上下文：数据作为 URL 参数，如 `<a href="/search?q=数据">`。此时需要进行 URL 编码（百分号编码，如 `%20`）

2. 什么是信任边界以外（Outside the trust boundary）：

   * 用户直接输入：表单、URL 参数、Cookie

   * 第三方系统：通过 API 获取的外部数据

   * 数据库/文件系统：即使是存入数据库的数据，在取出并显示时也应视为不可信，因为该数据可能在入库时未经过充分过滤，或者数据库本身已被污染

3. 不要试图使用一种编码方式解决所有问题



### 1.3.3 身份验证和密码管理

1. 除了那些特定设为“公开”的内容以外，对所有的网页和资源要求身份验证
2. 所有的身份验证过程必须在可信系统（比如：服务器）上执行
3. 在任何可能的情况下，建立并使用标准的、已通过测试的身份验证服务。为所有身份验证控制使用一个集中实现的方法，其中包括利用库文件请求外部身份验证服务
4. 只有当所有的数据输入以后，才进行身份验证数据的验证
5. 身份验证的失败提示信息应当避免过于明确。比如：可以使用“用户名和/或密码错误”，而不要使用“用户名错误”或者“密码错误”。错误提示信息在显示和源代码中应保持一致
6. 如果您的应用程序管理着凭证的存储，那么应当保证只保存了通过使用强加密单向 salted 哈希算法得到的密码（如果可以避免的话，不要使用 MD5 算法），并且只有应用程序具有对保存密码和密钥的表/文件的写权限。密码哈希必须在可信系统（比如：服务器）上执行
7. 用于访问应用程序以外服务的身份验证凭据信息应当加密，并存储在一个可信系统（比如：服务器）中受到保护的地方。源代码不是一个安全的地方
8. 通过政策或规则加强密码复杂度的要求（比如：要求使用字母、数字和/或特殊符号）。身份验证的凭据信息应当足够复杂以对抗在其所部署环境中的各种威胁攻击
9. 通过政策和规则加强密码长度要求。常用的是 8 个字符长度，但是 16 个字符长度更好，或者考虑使用多单词密码短语
10. 当连续多次登录失败后（比如：通常情况下是 5 次），应强制锁定账户。账户锁定的时间必须足够长，以阻止暴力攻击猜测登录信息，但是不能长到允许执行一次拒绝服务攻击
11. 所有的身份验证控制应当安全的处理未成功的身份验证
12. 用户账号的上一次使用信息（成功或失败）应当在下一次成功登录时向用户报告
13. 根据政策或规则的要求，强制定期更改密码。关键系统可能会要求更频繁的更改。更改时间周期必须进行明确
14. 密码重设和更改操作需要类似于账户创建和身份验证的同样控制等级
15. 当密码重新设置时，通知用户
16. 阻止密码重复使用。密码在被更改前应当至少使用了一天，以阻止密码重用攻击
17. 更改所有厂商提供的默认用户 ID 和密码，或者禁用相关帐号
18. 在执行关键操作以前，对用户再次进行身份验证
19. 为高度敏感或重要的交易账户使用多因子身份验证机制

注：

1. 多单词密码短语（multi-word pass phrases）：它不是一个单词，而是一串不相关的单词组合。例如：`Correct-Horse-Battery-Staple` 或 `Summer-Rain-Coffee-Piano-2026`。优势：
   - 易记性：用户更容易记住一段有画面感的短语，而不是乱码
   - 极高强度：这类短语通常长度超过 20 个字符，对于现有的攻击手段来说，破解难度极高
   - 防止字典攻击：由于单词组合是随机的，预计算的彩虹表（Rainbow table）几乎无法覆盖
2. 账户锁定策略中的平衡艺术：
   * 对抗暴力破解：如果连续 5 次失败之后锁定 30 分钟，攻击者尝试 100 个密码就需要耗费 10 个小时，这让自动化攻击变得毫无效率
   * 防止 DoS 攻击：如果你的锁定机制太严苛（例如错 5 次就永久锁定），攻击者只需要输入你的邮箱/用户名，然后乱打 5 次密码，你的账号就被锁死了。攻击者可以编写脚本，遍历全站用户名进行错误登录，导致全站合法用户都无法进入系统。这就是典型的针对业务逻辑的拒绝服务攻击
   * 进阶方案：锁 IP 而非锁用户名；引入 CAPTCHA（验证码），增加机器人的攻击成本
3. 阻止密码重用攻击（prevent attacks on password re-use）：这里主要指的是防止用户通过快速、多次地更新密码从而绕过“禁止重复使用最近 N 次旧密码”的限制



### 1.3.4 会话管理

1. 会话标识符必须总是在一个可信系统（比如：服务器）上创建
2. 会话管理控制应当使用通过审查的算法以保证足够的随机会话标识符
3. 在平衡的风险和业务功能需求的基础上，设置一个尽量短的会话超时时间。通常情况下，应当不超过几个小时
4. 如果一个会话在登录以前就建立，在成功登录以后，关闭该会话并创建一个新的会话。在任何重新身份验证过程中建立一个新的会话标识符
5. 生成一个新的会话标识符并周期性地使旧会话标识符失效（这可以缓解那些原标识符被获得的特定会话劫持情况）
6. 不允许同一用户 ID 的并发登录
7. 不要在 URL、错误信息或日志中暴露会话标识符。会话标识符应当只出现在 HTTP cookie 头信息中。比如，不要将会话标识符以 GET 参数进行传递
8. 为在 TLS 连接上传输的 cookie 设置 `secure` 属性。将 cookie 设置为 `HttpOnly` 属性，除非在应用程序中明确要求了客户端脚本程序读取或者设置cookie 的值
9. 注销功能应当完全终止相关的会话或连接

注：

1. 会话标识符应当只出现在 HTTP cookie 头信息中：OWASP 的这条规范写于 2010 年，当时主要是传统的 Cookie-Session 模式。对于现代 Web 开发来说，更常见的是使用 Auth Token 或者 JWT，这些可以放在自定义请求头中（例如：`Authorization: Bearer <token>`）进行传输而不是放在 cookie 请求头中，但是需要使用 HTTPS



### 1.3.5 访问控制

1. 只使用可信系统对象（比如：服务器端会话对象）以做出访问授权的决定
2. 安全的处理访问控制失败的操作
3. 将有特权的逻辑从其他应用程序代码中隔离开
4. 限制只有授权的用户才能访问服务器上的资源，包括：
   *  文件或其他资源
   * 受保护的 URL
   * 受保护的功能
   * 直接对象引用
   * 服务
   * 应用程序数据
   * 用户、数据属性和策略信息
   * 与安全相关的配置信息
5. 服务器端执行的访问控制规则和表示层实施的访问控制规则必须匹配
6. 限制单一用户或设备在一段时间内可以执行的事务数量。事务数量/时间应当高于实际的业务需求，但也应该足够低以判定自动化攻击
7. 如果长的身份验证会话被允许，周期性的重新验证用户的身份，以确保他们的权限没有改变。如果发生改变，注销该用户，并强制他们重新执行身份验证
8. 执行帐户审计并将没有使用的帐号强制失效（比如：在用户密码过期后的 30 天以内）
9. 应用程序必须支持帐户失效，并在帐户停止使用时终止会话（比如：角色、职务状况、业务处理的改变，等等）

注：

1. 表示层实施的访问控制规则（presentation layer representations of access control rules）：这里指前端的目录、菜单、按钮根据用户菜单权限的展示规则



### 1.3.6 加密规范

1. 所有用于保护来自应用程序用户秘密信息的加密功能都必须在一个可信系统（比如：服务器）上执行
2. 保护主密钥（master secrets）信息免受未授权的访问
3. 安全的处理加密模块失败的操作
4. 为防范对随机数据的猜测攻击，应当使用加密模块中已验证的随机数生成器生成所有的随机数、随机文件名、随机 GUID 和随机字符串
5. 应用程序使用的加密模块应当遵从 FIPS 140-2 或其他等同的标准（请见：http://csrc.nist.gov/groups/STM/cmvp/validation.html）

注：

1. 第 5 点更强调我们在技术选型时优先选择已经获得认证的加密提供程序（Crypto Provider）



### 1.3.7 错误处理和日志

1. 不要在错误响应中泄露敏感信息，包括：系统的详细信息、会话标识符或者帐号信息。不要在日志中保存敏感信息，包括：不必要的系统详细信息、会话标识符或密码
2. 使用错误处理以避免显示调试或堆栈跟踪信息
3. 使用通用的错误消息并使用定制的错误页面
4. 所有的日志记录控制应当在可信系统（比如：服务器）上执行
5. 日志记录控制应当支持记录特定安全事件的成功或者失败操作
6. 确保日志记录中包含的不可信数据，不会在查看界面或者软件时以代码的形式被执行
7. 应该通过日志记录这些事件：
   * 所有失败的输入验证
   * 所有的身份验证尝试，特别是失败的验证
   * 所有失败的访问控制
   * 明显的修改事件，包括对于状态数据非期待的修改
   * 连接无效或者已过期的会话令牌尝试
   * 所有的系统例外
   * 所有的管理功能行为，包括对于安全配置设置的更改
   * 所有失败的后端 TLS 链接
   * 加密模块的错误
8. 确保一个执行日志查询分析机制的存在
9. 使用加密哈希功能以验证日志记录的完整性

注：

1. 通用的错误信息（generic error messages）：意味着无论发生什么具体的后台错误，展示给外部用户的信息都是模糊且一致的
   * 正面例子（通用化处理）：
     1. `系统繁忙，请稍后再试`（通用的 500 错误）
     2. `用户名或密码错误`（不指明具体哪部分错了）
   * 反面例子（泄露敏感信息）：
     1. `Error 500: Connection failed to database 'UserDB' on port 5432.`（泄露数据库名和端口）
     2. `Invalid login: User 'admin' exists but password incorrect.`（泄露用户名是否存在，有助于暴力破解）



### 1.3.8 数据保护

1. 授予最低权限，以限制用户只能访问为完成任务所需要的功能、数据和系统信息
2. 即使在服务器端，任然要加密存储的高度机密信息，比如，身份验证的验证数据。总是使用已经被很好验证过的算法
3. 保护服务器端的源代码不被用户下载
4. 删除用户可访问产品中的注释，以防止泄露后台系统或者其他敏感信息
5. 删除不需要的应用程序和系统文档，因为这些也可能向攻击者泄露有用的信息
6. 不要在 HTTP GET 请求参数中包含敏感信息
7. 应用程序应当支持，当数据不再需要的时候，删除敏感信息（比如：个人信息或者特定财务数据）



### 1.3.9 通讯安全

1. 为所有敏感信息采用加密传输。其中应该包括使用 TLS 对连接的保护，以及支持对敏感文件或非基于 HTTP 连接的不连续加密
2. 为所有要求身份验证的访问内容和所有其他的敏感信息提供 TLS 连接
3. 没有成功的 TLS 连接不应当后退成为一个不安全的连接



### 1.3.10 系统配置

1. 确保服务器、框架和系统部件采用了认可的最新版本。确保服务器、框架和系统部件安装了当前使用版本的所有补丁
2. 关闭目录列举功能
3. 将 Web 服务器、进程和服务的账户限制为尽可能低的权限
4. 移除所有不需要的功能和文件
5. 通过将不进行对外检索的路径目录放在一个隔离的父目录里，以防止目录结构在 robots.txt 文档中暴露。然后，在 robots.txt 文档中“禁止”整个父目录，而不是对每个单独目录的“禁止”
6. 移除在 HTTP 相应报头中有关 OS、Web 服务版本和应用程序框架的无关信息
7. 将开发环境从生产网络隔离开，并只提供给授权的开发和测试团队访问。开发环境往往没有实际生产环境那么安全，攻击者可以使用这些差别发现共有的弱点或者是可被利用的漏洞

注：

1. robots.txt：robots.txt 的初衷是告诉合法的搜索引擎爬虫（如 Googlebot）哪些页面不应该被索引。然而，它是一个所有人都能公开访问的文件，如果你在其中列出了大量具体目录，比如 `Disallow: /admin_backend_v2/` 或 `Disallow: /config_backup_2026/`，你实际上是在给攻击者提供一张“攻击指南” ，攻击者甚至不需要扫描你的服务器，只要看一眼 robots.txt 就能知道你的敏感目录在哪里



### 1.3.11 数据库安全

1. 使用强类型的参数化查询方法（即使用`prepared statements`）
2. 确保变量是强类型的
3. 当应用程序访问数据库时，应使用尽可能最低的权限
4. 为数据库访问使用安全凭证
5. 连接字符串不应当在应用程序中硬编码。连接字符串应当存储在一个可信服务器的独立配置文件中，并且应当被加密
6. 删除或者修改所有默认的数据库管理员密码。使用强密码、强短语，或者使用多因子身份验证
7. 关闭所有不必要的数据库功能（比如：不必要的存储过程或服务、应用程序包、仅最小化安装需要的功能和选项（表面范围缩减））
8. 删除厂商提供的不必要的默认信息（比如：数据库模式示例）
9. 应用程序应当以不同的凭证为每个信任的角色（比如：用户、只读用户、访问用户、管理员）连接数据库



### 1.3.12 文件管理

1. 不要把用户提交的数据直接传送给任何动态调用功能（例如`new File("/user-input-filepath")`）
2. 在允许上传一个文档以前进行身份验证
3. 只允许上传满足业务需要的相关文档类型。通过检查文件报头信息，验证上传文档是否是所期待的类型。只验证文件类型扩展是不够的
4. 不要把文件保存在与应用程序相同的 Web 环境中。文件应当保存在内容服务器或者数据库中
5. 防止或限制上传任意可能被 Web 服务器解析的文件
6. 关闭在文件上传目录的运行权限
7. 不要传递目录或文件路径，使用预先设置路径列表中的匹配索引值
8. 绝对不要将绝对文件路径传递给客户
9. 确保应用程序文件和资源是只读的
10. 对用户上传的文件扫描进行病毒和恶意软件



### 1.3.13 内存管理

1. 如果在循环中调用函数时，检查缓存大小，以确保不会出现超出分配空间大小的危险
2. 在将输入字符串传递给拷贝和连接函数前，将所有输入的字符串缩短到合理的长度
3. 关闭资源时要特别注意，不要依赖垃圾回收机制（比如：连接对象、文档处理，等）

# 第二部分：OWASP Top 10:2025

> 链接：https://owasp.org/Top10/2025/A01_2025-Broken_Access_Control/

OWASP 官方在 2025 年发布了 `OWASP Top 10:2025`（上一期是在 2021 年发布的），2025 版 Top 10 排名如下：

1. `A01 Broken Access Control`（访问控制策略故障）
2. `A02 Security Misconfiguration`（安全配置错误）
3. `A03 Software Supply Chain Failures`（软件供应链故障）
4. `A04 Cryptographic Failures`（加密失败）
5. `A05 Injection`（注入）
6. `A06 Insecure Design`（不安全的设计）
7. `A07 Authentication Failures`（身份认证失败）
8. `A08 Software or Data Integrity Failures`（软件或数据完整性故障）
9. `A09 Security Logging & Alerting Failures`（安全日志记录和警报故障）
10. `A10 Mishandling of Exceptional Conditions`（异常情况处理不当）

下面将聚焦于其中那些比较新颖、或者容易被疏忽的漏洞进行说明

## 2.1 A01 Broken Access Control

1. 特权提升，获得超出登录用户预期的权限（例如管理员访问权限）
2. 通过修改 URL（参数篡改或强制浏览）、内部应用程序状态或 HTML 页面，或者使用修改 API 请求的攻击工具来绕过访问控制检查

示例：演示HUB运营平台普通账号访问角色管理页面（http://op-hub-dev.xgd.com/role）时可以获取到整个平台菜单权限数据的例子



## 2.2 A02 Security Misconfiguration

1. 启用或安装了不必要的功能（例如，不必要的端口、服务、页面、帐户、测试框架或权限）
2. 对于升级的系统，最新的安全功能被禁用或未安全配置



## 2.3 A03 Software Supply Chain Failures

1. 没有仔细跟踪您使用的所有组件（客户端和服务器端）的版本。这包括您直接使用的组件以及嵌套（传递）依赖项
2. 没有定期扫描漏洞并订阅与您使用的组件相关的安全公告

近年来第三方组件、依赖、工具投毒事件屡见不鲜



## 2.4 A07 Authentication Failures

1. 凭证填充，即使用已知用户名和密码组合的列表，现在是一种非常常见的攻击。最近，攻击者被发现根据常见的人类行为“增加”或以其他方式调整密码。例如，将“Winter2025”更改为“Winter2026”，或将“ILoveMyDog6”更改为“ILoveMyDog7”或“ILoveMyDog5”。这种对密码尝试的调整称为混合凭证填充攻击或密码喷射攻击，它们甚至比传统版本更有效。如果应用程序没有针对自动威胁（暴力、脚本或机器人）或凭证填充实施防御，则该应用程序可以用作密码预言机来确定凭证是否有效并获得未经授权的访问



## 2.5 A09 Security Logging & Alerting Failures

1. 可审核事件（例如登录、登录失败和高价值事务）不会被记录或记录不一致（例如，仅记录成功的登录，但不记录失败的尝试）
2. 不监控应用程序和 API 日志中的可疑活动
3. 应用程序丢失或错误处理错误以及其他异常情况，导致系统不知道存在错误，因此无法记录存在问题
4. 太多误报警报导致无法区分重要警报和不重要警报，从而导致识别得太晚或根本无法识别（SOC 团队的体力超负荷）
5. 行为分析和人工智能支持可以作为可选的附加技术，以支持警报的低误报率

# 第三部分：OWASP for AI

## 3.1 OWASP GenAI Security Project

> 链接：https://genai.owasp.org/

OWASP 正在从“经典 Web 安全知识库”扩展成“现代软件与 AI 系统安全体系”：

- 从 Web 应用扩展到 LLM、Agent、MCP、Skill
- 从编码缺陷扩展到协议、身份、供应链、治理

OWASP GenAI Security Project 的成立说明 OWASP 不再把 AI 安全理解成“给 LLM 列 10 条风险”这么简单，而是开始建设一整套覆盖当前生成式 AI 各个方面的持续安全治理体系

## 3.2 LLM Top 10:2025

> 链接：https://genai.owasp.org/llm-top-10/

如果只把 AI 风险理解成“幻觉”，那会把问题看窄很多。按 OWASP 官方 `LLM Top 10:2025` 的口径，所谓“幻觉”更接近 `LLM09 Misinformation` 这一个风险点里的部分表现，但真正的安全面已经同时覆盖：

- 输入会不会操纵模型
- 敏感数据会不会被模型吐出来
- 模型、数据、LoRA、依赖和平台会不会被供应链污染
- 模型输出会不会直接触发下游危险动作
- Agent 会不会因为权限过大而替攻击者做事
- 向量库、Embedding、RAG 会不会变成新的泄露面
- token、算力、费用会不会被无限消耗

### LLM01:2025 - Prompt Injection

> 提示词注入

- 是什么：攻击者通过用户输入、网页、文件、图片等内容改变模型行为，让模型偏离原本指令。
- 为什么：它可能导致越权调用工具、泄露系统提示词、泄露数据、操纵决策，甚至驱动下游系统执行危险动作。
- 怎么防：约束模型职责边界，隔离外部内容，严格校验输入输出，对高风险动作做人审或二次确认，不把安全控制只寄托在 prompt 上。

### LLM02:2025 - Sensitive Information Disclosure

> 敏感信息泄露

- 是什么：模型在输出中泄露个人信息、业务敏感信息、凭据、训练数据片段、内部配置或专有知识。
- 为什么：LLM 会把训练数据、上下文数据、RAG 检索结果和系统信息混在一起生成回答，一旦边界没控好，就会把“不该说的”说出来。
- 怎么防：做脱敏和红线识别，限制模型可见数据范围，避免默认把用户输入用于训练，清晰声明数据使用方式，并对输出做敏感信息检测。

### LLM03:2025 - Supply Chain

> 供应链风险

- 是什么：风险不只来自业务代码，也来自模型、数据集、向量库、LoRA 适配器、开源依赖、镜像、训练平台和推理平台。
- 为什么：第三方模型和数据很难像普通代码一样完全审计，一旦被投毒、篡改或替换，模型行为就可能整体失真，甚至带入后门。
- 怎么防：只用可信来源，维护 SBOM/AIBOM 或模型清单，做签名和哈希校验，定期补丁与重评估，对供应商、模型和数据做安全评估与红队验证。

### LLM04:2025 - Data and Model Poisoning

> 数据和模型投毒

- 是什么：攻击者污染预训练、微调、Embedding 或知识库数据，让模型学到偏置、后门或错误事实。
- 为什么：这会直接影响模型完整性，让模型在特定触发条件下输出被操纵的结果。
- 怎么防：跟踪数据来源和版本，严格筛选训练与知识库数据，对异常样本做检测，做鲁棒性测试和红队验证。

### LLM05:2025 - Improper Output Handling

> 输出处理不当

- 是什么：系统把 LLM 输出当成可信内容，未经校验就传给页面、脚本、SQL、命令、邮件模板或其他下游系统。
- 为什么：LLM 输出本质上仍然是不可信输入，只是来源从“用户”变成了“模型”，一样可能引发 XSS、注入、命令执行或流程劫持。
- 怎么防：把模型当成不可信协作者，对输出做校验、编码和白名单约束，高风险下游只接受结构化和可验证格式。

### LLM06:2025 - Excessive Agency

> 代理权限过大

- 是什么：给 LLM 或 Agent 过大的工具权限、数据权限和执行权限，让它能“直接做事”而不是“只负责建议”。
- 为什么：一旦模型被误导、越狱或理解错误，过大的 agency 会把问题从“说错”升级成“做错”。
- 怎么防：最小权限、任务级授权、敏感操作审批、工具分级、审计日志和可撤销机制。

### LLM07:2025 - System Prompt Leakage

> 系统提示词泄露

- 是什么：系统提示词、本地规则、隐藏指令或其中包含的敏感信息被模型泄露给用户。
- 为什么：很多团队误以为 system prompt 天然保密，但它常常能被套问、抽取、拼接或侧信道方式暴露。
- 怎么防：不要把密钥、凭据、权限规则藏在 prompt 里；把关键控制放到确定性代码里；减少 prompt 中的敏感内容并做隔离。

### LLM08:2025 - Vector and Embedding Weaknesses

> 向量库和嵌入式弱点

- 是什么：向量库、Embedding、RAG 检索链路在权限隔离、数据完整性、检索准确性上存在缺陷。
- 为什么：它会导致跨租户检索、知识库投毒、错误召回和隐私泄露，尤其在多租户和企业知识库场景里非常现实。
- 怎么防：对向量数据做权限隔离和来源治理，控制写入链路，监控检索质量，对知识库变更做审核和追踪。

### LLM09:2025 - Misinformation

> 错误信息

- 是什么：模型生成看似可信但错误、偏置、误导甚至危险的信息。
- 为什么：它会影响用户判断、业务决策、客服答复、医疗金融建议和代码生成质量，所以“幻觉”本身也会演变成安全与合规问题。
- 怎么防：高风险场景引入事实校验、来源引用、人工复核、领域限制和明确免责声明，不让模型单独承担关键决策。

### LLM10:2025 - Unbounded Consumption

> 无限制消耗

- 是什么：攻击者通过超长输入、高频请求、复杂查询或模型提取行为，持续消耗 token、算力、费用和服务能力。
- 为什么：LLM 的成本和资源消耗远高于普通接口，所以这类问题会同时带来 DoS、拒绝钱包和模型能力被“薅走”的风险。
- 怎么防：限流、配额、超时、输入长度限制、资源监控、异常检测、优雅降级和高成本操作隔离。



## 3.3 Agentic Skills Top 10

> https://owasp.org/www-project-agentic-skills-top-10/

OWASP 官方对 skill 的定位非常清楚：它不是单个工具，不只是 prompt 片段，而是一种可复用的“行为模板”或“工作流模板”。它往往会同时定义：

- 任务拆解方式
- 工具调用顺序
- 文件、网络、Shell 的使用方式
- 安全护栏和输出格式
- 持久记忆和跨会话状态

这也是它为什么值得单独列一个 Top 10。因为 skill 出问题时，被影响的已经不是“回答质量”，而是：

- agent 会不会被教成恶意行为
- 工具权限会不会被放大
- 文件和凭据会不会被读走
- 自动化流程会不会被篡改
- 风险会不会跨平台复用和扩散

### AST01 - Malicious Skills

> 恶意技能

- 是什么：攻击者发布看起来正常、实则内含恶意载荷的 skill，可能在说明文档、配置、脚本或依赖里藏后门。
- 为什么：skill 往往继承 agent 的真实权限，一旦安装成功，就可能直接接触 SSH key、API key、浏览器数据、钱包文件、Shell 和网络能力。
- 怎么防：只允许可信发布源，要求签名和注册表扫描，在发布和安装时做行为分析，把 skill 运行放进沙箱，并建立信誉与审计机制。

### AST02 - Supply Chain Compromise

> 供应链风险

- 是什么：skill 注册表、发布流程、依赖树、配置文件和维护者账号被污染，导致看似正常的 skill 通过供应链进入环境。
- 为什么：skill 生态还没有像 npm、PyPI 那样成熟的来源证明和透明日志，一旦上游被劫持，下游 agent 会直接继承被污染结果。
- 怎么防：做来源证明、透明日志、递归依赖扫描、哈希锁定和内部 allowlist，把 skill 配置文件也当成可执行面来审查。

### AST03 - Over-Privileged Skills

> 过度授权的Skills

- 是什么：skill 拿到的文件、网络、数据库、Shell、记忆文件写入等权限，明显大于它完成任务实际所需。
- 为什么：一旦 skill 被 prompt injection、误操作或恶意输入驱动，过大权限会把本来局部的问题放大成整体破坏。
- 怎么防：要求声明权限清单，使用最小权限和按 skill 隔离的凭据，对运行时真实行为做校验，并对高危权限做额外审核。

### AST04 - Insecure Metadata

> 不安全的元数据

- 是什么：skill 的名称、作者、权限声明、风险等级、依赖说明等元数据可被伪造、弱化或伪装。
- 为什么：用户和平台常常首先相信元数据，一旦它被操纵，就会出现品牌冒充、权限低报、风险误导和搜索投毒。
- 怎么防：对元数据做静态分析和格式校验，对品牌和发布者做校验，把声明权限与真实行为比对，并展示元数据来源与签名信息。

### AST05 - Unsafe Deserialization

> 不安全的反序列化

- 是什么：skill 加载过程会解析 YAML、JSON、Markdown、依赖文件等，如果解析器或加载流程不安全，可能在加载时就执行恶意内容。
- 为什么：skill 的攻击面不只在运行阶段，也在“被安装、被解析、被初始化”的那一刻，而且往往是自动发生的。
- 怎么防：使用安全解析器，限制允许字段，在隔离环境中解析和安装依赖，把 skill 文件当成不可信输入而不是静态配置。

### AST06 - Weak Isolation

> 弱隔离性

- 是什么：skill 与宿主 agent 运行在同一安全上下文，直接共享主机文件系统、网络、Shell 和长期凭据。
- 为什么：没有隔离时，任何一个 skill 的问题都可能直接变成整机问题，沙箱、容器、命名空间这些边界等于不存在。
- 怎么防：默认容器化或沙箱化执行，限制宿主模式，隔离每个 skill 的进程和资源，并对本地控制接口做认证、限流和最小暴露。

### AST07 - Update Drift

> 更新漂移

- 是什么：已安装的 skill 长期不更新，或自动更新机制盲目信任上游，导致要么补丁滞后，要么恶意更新被自动吃进来。
- 为什么：skill 常由个人自行安装，缺少企业补丁管理；而且“新版本”本身如果没有签名和哈希，也未必值得信任。
- 怎么防：使用不可变哈希固定版本，对每次更新都做签名校验，生产环境禁用热更新，订阅安全公告并维护安装清单。

### AST08 - Poor Scanning

> 扫描不足

- 是什么：传统代码扫描器很难识别 skill 里的自然语言恶意意图、上下文依赖行为和隐藏在文档中的指令。
- 为什么：skill 既有代码层，又有说明层；攻击者完全可以不写危险代码，只用文字诱导 agent 去做危险动作。
- 怎么防：同时扫描代码层和自然语言层，用语义分析加行为沙箱的多工具流水线，不把单一扫描结果当成最终信任依据。

### AST09 - No Governance

> 治理缺失

- 是什么：组织里没有 skill 资产清单、审批流程、审计日志、撤销机制和责任归属，skill 安装与使用处于“影子 AI”状态。
- 为什么：安全团队看不见 skill，运维流程管不到 skill，离职、事故、审计、合规都很难收口。
- 怎么防：建立集中 inventory、审批和审计，把 skill 接入 CMDB/ITSM/CASB，对 agent 使用非人身份和生命周期管理。

### AST10 - Cross-Platform Reuse

> 跨平台复用风险

- 是什么：同一个 skill 在 OpenClaw、Claude Code、Cursor、VS Code 等平台间迁移时，安全属性没有被等价保留。
- 为什么：不同平台的 skill 格式、权限模型和风险字段不统一，迁移时很容易把关键安全信息“翻译丢了”。
- 怎么防：迁移时重新做安全校验，推动统一 skill 格式或字段映射，建立跨注册表威胁情报共享和平台无关的扫描能力。
